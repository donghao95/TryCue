# 09 部署与运维

本文档涵盖部署、环境变量、模型配置、可观测性与排障。

## 1. 目标

本文档定义 AI 用户试映会 V1 的本地开发、环境变量、部署配置、运行参数、日志追踪与排障方案。

V1 建议：

```text
Vite React 前端
Fastify API 后端
SQLite 本地库（未来生产可迁移 PostgreSQL）
OpenAI-compatible Chat Completions
SSE
内存 Scheduler
```

## 5. 本地启动流程

推荐 Windows / PowerShell 一键启动：

```powershell
pnpm dev:local
```

该脚本会检查 Docker daemon（可选），补齐 `.env.local`、生成 Prisma Client、执行数据库迁移，然后同时启动 API 和 Web。当前本地数据库为 SQLite，无需 Docker。跳过 Docker 检查可使用：

```powershell
powershell -ExecutionPolicy Bypass -File ./scripts/run-local.ps1 -SkipDocker
```

手动启动流程：

```bash
pnpm install
cp .env.example .env.local
pnpm db:migrate
pnpm dev
```

本地 API 启动时会自动从工作区根目录加载 `.env.local` 和 `.env`。`.env.local` 优先，`.env` 只补充缺失项；已经由 shell 或部署平台提供的环境变量不会被覆盖。LLM 的 `runtimeMode`、`apiKey`、`baseUrl` 和模型选择不写入 `.env`，仍由 `config/llm.local.yaml` 作为唯一运行时配置源。

本地开发库和集成测试库必须隔离：

```text
本地开发：file:./data/trycue.db（SQLite）
集成测试：file:./data/trycue_test.db（SQLite）
```

`pnpm test:integration` 会使用 `file:./data/trycue_test.db` 执行迁移，并以 `NODE_ENV=test` 运行完整 API 集成测试。`pnpm test:integration:smoke` 使用同一测试库和测试环境，只运行 PR CI 需要的关键 smoke 子集。测试清库 helper 会拒绝重置任何 `DATABASE_URL` 不包含 `test` 的库，避免测试数据污染本地开发历史列表。

Mock agent 展示延迟通过 `MOCK_AGENT_DELAY_MS` 环境变量控制，默认 0（无延迟），test 和 dev 行为一致。本地 demo 需要看实时过程时，显式设置 `MOCK_AGENT_DELAY_MS=2000` 等值即可。注意：所有 mock 调用（profile 展开、audience turn、sampling plan）使用同一延迟值，不再区分阶段节奏；若需模拟原分阶段延迟，需在代码层调整。

启动后访问：

```text
http://localhost:3000
```

当前实现采用 Vite 单页工作台作为第一屏，创建、启动和实时观看都在根页面完成；API 服务默认运行在 `http://localhost:4000`，Vite 通过 dev server proxy 转发 `/api` 和 `/uploads`。

本地上传文件固定保存到工作区根目录下的 `apps/api/uploads`。路径解析不得依赖启动命令的当前工作目录。

## 9. 模型配置

LLM 运行配置不再放入 `.env`，而是使用 YAML 文件作为唯一配置源。`.env` 只可通过 `LLM_CONFIG_PATH` 指定文件路径。

开发环境 `config/llm.local.yaml`：

```yaml
provider: openai-compatible
runtimeMode: mock
apiKey: ""
baseUrl: ""
models:
  fast: ""
  pro: ""
```

真实大模型运行：

```yaml
provider: openai-compatible
runtimeMode: real
apiKey: <用户自行填写>
baseUrl: <OpenAI-compatible base URL，真实模型运行必填>
models:
  fast: <用户自行填写>
  pro: <用户自行填写，需支持 vision>
```

实际模型由用户按成本、速度、视觉能力和质量自行选择。文档不固定具体模型名，避免实现时误以为示例值就是默认配置。

### 9.1 图片与上传限制

```text
上传接口单张图片最大文件大小：5 MB（MAX_COVER_IMAGE_SIZE_MB）
前端选择超过 5 MB 的图片时，会在上传前压缩到 5 MB 以内且最长边不超过 1600px；后端拒绝超过 5 MB 或最长边超过 4096px 的图片
单篇内容最多 9 张图片，第一张兼容为 coverImageUrl
存储方式：本地文件系统（apps/api/uploads 目录，由 @fastify/static 服务）
图片访问 URL：/uploads/<filename>
允许的图片格式：jpg, jpeg, png, webp
```

### 9.2 多模态模型配置

Pro 模型需使用多模态模型（支持 vision），用于采样计划、优化问答、打磨人设和报告中的封面图等视觉内容：

```yaml
provider: openai-compatible
runtimeMode: real
apiKey: <用户自行填写>
baseUrl: <OpenAI-compatible base URL，真实模型运行必填>
models:
  fast: <用户自行填写>
  pro: <用户自行填写，需支持 vision>
```

Pro 模型需支持 vision 能力，以实现对真实封面图的多模态分析；后端不会自动降级到快速模型。

真实 AI run 的配置规则：

```text
1. runtimeMode=mock 时使用 Mock provider，不要求 API key、baseUrl 或模型配置；即使这些字段完整也不会调用真实模型。
2. runtimeMode=real 时必须同时提供 apiKey、baseUrl、models.fast、models.pro，保存接口会做校验。
3. baseUrl 必须是 OpenAI-compatible 网关地址，例如以 /v1 结尾的服务地址；只限制 http / https scheme，不禁止 127.0.0.1、localhost、私网或局域网地址。
4. real run 遇到 base URL 配置错误、模型不支持 vision 或模型调用失败时，API 返回 CONFIG_ERROR 或 AGENT_RUN_FAILED。
5. 缺少 runtimeMode 的 YAML 配置不兼容，服务启动时应失败而不是推断模式。
6. 设置接口不再接受 execution 配置；模型调用策略由轻量 AI Task Runner 内置的 taskType 映射决定。
```

### 9.3 本地运行时模型设置

本地 V1 提供 `/settings` 页面和以下 API，便于在不重启前端页面的情况下配置大模型：

```text
GET /api/settings/llm
PUT /api/settings/llm
POST /api/settings/llm/models
GET /api/settings/llm/capacity/status
POST /api/settings/llm/capacity/probe
GET /api/settings/llm/capacity/probe/:jobId
POST /api/settings/llm/capacity/probe/:jobId/cancel
POST /api/settings/llm/capacity/reset-learning
POST /api/settings/llm/capacity/apply-recommended
```

设计边界：

```text
这些接口是本地单用户 V1 的管理能力，当前版本刻意不在应用内实现账号体系、权限校验或接口鉴权。
该设计只适用于 localhost、受信任局域网或已经由外部网关保护的部署。
它不是可直接裸露到公网的默认安全模型，也不代表未来公开部署版本可以继续保持无鉴权。
```

运行时设置保存到 `config/llm.local.yaml`，并在保存成功后立即更新当前 API 服务进程内快照。服务重启后重新读取同一个 YAML 文件。

`.env` 中的 `LLM_REQUESTS_PER_MINUTE` 已完整移除，不保留向后兼容 fallback；旧版本升级时该环境变量被忽略，限流配置以 `config/llm.local.yaml` 的 `capacity` 字段为唯一来源。

容量预设数值参考（`LLM_CAPACITY_PRESET_VALUES`）：

| 预设 | initialRpm | minRpm | maxRpm | initialConcurrency | minConcurrency | maxConcurrency | 适用场景 |
|---|---:|---:|---:|---:|---:|---:|---|
| `conservative` | 4 | 1 | 30 | 2 | 1 | 2 | 低额度、未知网关或经常触发限流 |
| `standard` | 8 | 2 | 60 | 4 | 1 | 4 | 默认，适合普通 OpenAI-compatible 网关 |
| `high_quota` | 30 | 5 | 300 | 8 | 1 | 16 | 用户明确知道账号额度较高 |
| `custom` | 用户填写 | 用户填写 | 用户填写 | 用户填写 | 用户填写 | 用户填写 | 高额度场景，受 hardMaxRpm/hardMaxConcurrency 保护 |

`hardMaxRpm` / `hardMaxConcurrency` 是绝对保护上限（默认 1000 / 100），防止误配置或探测失控。切换 preset 时前端立即更新表单草稿，但不自动保存。

安全要求：

```text
API key 不从 GET 接口回显
API key 不写入前端 localStorage、URL 或日志
模型列表通过服务端请求 OpenAI-compatible /models，前端只接收模型 id
模型 baseUrl 可指向本机、私网或局域网网关；公网部署时必须用访问控制、鉴权、反向代理或内网管理入口限制设置接口和模型列表接口调用方
```

风险提示：

```text
如果将当前设置接口裸露到不可信网络，未授权调用方可以修改服务端 LLM 配置，或诱导服务端携带所提供/已保存的 API key 请求任意 http(s) 模型地址。
这可能造成密钥滥用、配置篡改、费用损失，以及对 localhost、私网或局域网服务的服务端请求风险。
公网部署必须在请求到达 TryCue API 之前完成身份认证和访问控制；仅配置 CORS 不能替代服务端鉴权。
```

## 10. 超时配置

AgentRunner 采用分层 timeout：

```text
MODEL_CALL_TIMEOUT_SECONDS          单次模型调用超时（默认 600 秒）
MODEL_STEP_TIMEOUT_SECONDS          单个 Agent step 超时（默认 300 秒）
AGENT_JOURNEY_TIMEOUT_SECONDS       单个 Agent 从开始运行到 terminal 的总时长上限（默认 600 秒）
RUNNER_HEARTBEAT_INTERVAL_SECONDS   单机内存 runner 的心跳更新间隔（默认 5 秒）
RUN_CLOCK_SCALE                     模拟时钟与真实时间的缩放比（默认 10）
SSE 心跳间隔：15 秒（SSE_HEARTBEAT_INTERVAL）
```

采样计划、优化问答、打磨人设、画像展开、人设生成和 Report 走同一轻量 AI Task Runner。MVP Runner 负责 taskType、fast/pro 模型分流和基础调用记录，不提供全局 AI 并发池、排队或短总时长超时。Real provider 的非流式 `chat.completions.create` 调用可以对 429 / 408 / 5xx / 网络超时做短退避重试；重试耗尽后仍按原有 AGENT_RUN_FAILED 或局部失败路径处理。采样计划 streaming 生成和报告生成当前不套用这条短重试规则。AgentJourney 级别的运行保护由 AgentRunner 的分层 timeout 负责；模型调用和 tool call 的实际 timeout 会被当前 AgentJourney 剩余预算裁剪，模型调用超时会向 provider 传递 abort signal。

## 12. 生产环境注意事项

### 12.1 防止重复 Scheduler

如果部署到 Vercel Serverless，不适合长时间内存 Scheduler。

可选方案：

```text
1. V1 本地 / 单机 Node 服务部署。
2. 后续迁移到独立 worker。
3. 后续引入 Supabase Edge Function + cron。
4. 后续引入队列系统。
```

V1 为了快速验证，推荐单机服务。

### 12.2 SSE 连接限制

V1 只实现 SSE，不提供轮询降级。部署平台必须支持长连接；如果平台不稳定支持 SSE，应换单机 Node 服务或支持长连接的平台，而不是在 V1 引入轮询分支。

部署时需确保：
- 反向代理（如 Nginx）未对 SSE 连接做缓冲（需设置 `X-Accel-Buffering: no`）。
- 负载均衡支持长连接粘性会话（sticky session），避免 SSE 连接被路由到不同实例。
- SSE_HEARTBEAT_INTERVAL（默认 15 秒）根据平台限制调整。

（SSE 推送的完整 LiveUpdate 事件类型列表见下文第 13.8 节。）

### 12.3 API Key 安全

```text
config/llm.local.yaml 中的 apiKey 只在服务端使用
不要暴露到前端
```

### 12.4 成本控制

```text
默认快速试映 12 人用于开发
标准试映 30 人用于正式体验
限制正文最大长度
限制每个 journey 最大回合
限制每个 action 最大 tool calls
```

### 12.5 观众头像资源与许可

V1 使用开源统一风格头像（程序化 seed 生成或开源扁平插画头像），不使用 AI 实时生成头像。部署前需确认：

```text
1. 头像静态资源已打包或通过 CDN 部署，前端可正常加载。
2. 头像素材的开源许可允许商用（commercial use）。
3. 许可是否要求署名（attribution），如需署名需在产品适当位置标注。
4. 许可是否允许修改（modifications），V1 可能需要裁切或调整尺寸。
5. 许可是否允许作为产品 UI 素材使用（product UI usage）。
```

推荐头像来源优先级：程序化开源头像（基于 seed 生成固定头像）> 开源扁平插画头像 > 图标化角色头像。风格要求统一扁平、低饱和色、圆形裁切。

### 12.6 前端视觉主题

现场页采用浅色主题（浅灰白 / 米白 / 极淡紫灰背景），Apple / OpenAI 风格留白与克制，不使用深色控制台风格。部署时确保 CSS 静态资源正常加载，主题相关变量不被环境覆盖。

## 13. 可观测性与排障

AI 用户试映会是 Agent 驱动系统，必须可追溯、可回放、可排障。

### 13.1 追踪层级

```text
run
  └─ content_version
  └─ audience_agent
          └─ journey (agent_journeys)
              └─ agent_transcript_item (append-only transcript)
              └─ agent_turn (关联到 journey)
              └─ agent_tool_call
                  └─ interaction / comment / loaded_comment_page / action_log
```

### 13.2 必须记录的 ID

每条日志尽量包含：

```text
runId
contentVersionId
participantId
journeyId
sessionId (feed / post)
journeyActionId
toolCallId
sdkTraceId
promptVersion
model
llmCallTraceId
```

### 13.3 Action 调试信息

每个 `agent_turn` 应能查看：

```text
turn 基本信息（step_index、status）
所属 journey（通过 journey 的 content_version 推导 feed/post 阶段）
调用前上下文（agent_turn_contexts）
thought_text（通过 agent_transcript_items 记录）
available_tools（按阶段固定）
persona
Agent tool calls 原始结果
提交结果
LiveUpdates
错误信息
```

### 13.4 Tool Call 调试信息

每个 `agent_tool_call` 应能查看：

```text
tool_name
call_index
input
output
status
ignored / failed reason
simulated_time
```

### 13.5 评论追溯

点击一条评论，应能追溯：

```text
评论文本
回复对象 parent_comment_id
观众 persona
当前页面状态
当时可见的其他评论
本轮 thought_text
write_comment tool call
```

这样才能判断评论是否合理。

### 13.6 现场证据追溯

现场页主界面不再常驻展示完整行动日志。用户应通过观众详情、洞察详情或最终报告证据链追溯过程证据；调试页面仍可直接查看 action_logs。

点击一个观众或一个洞察，应能看到：

```text
相关观众 persona
旅程时间线
关键行为
本次评论
相关洞察
行动日志证据
```

每条行动日志证据应能看到：

```text
来源 thought_text 或关联 tool call
thought_text
关联动作
topic_tags
risk_tags
当时 screen
当时 post_state
```

注意：这里展示的是角色反应，不是模型隐藏推理链。

```text
run.pausing

run_control.event_recorded
```

### 13.8 SSE LiveUpdate 事件类型

现场页通过 SSE 推送以下 LiveUpdate 事件类型，驱动观众席状态矩阵、HUD 和模拟页的实时联动：

```text
post_state.updated          — 帖子状态变化（点赞/收藏/评论数更新）
comments.page_loaded        — 评论区页面加载完成
comment.created             — 新评论创建
audience.status_updated     — 观众状态变更（更新观众席角标）
audience.action_happened    — 观众关键行为发生（触发动画）
action_log.created          — 行动日志创建（仅用于观众详情/洞察详情证据链，不在主界面常驻展示）
summary.updated             — 实时摘要更新
insight.created             — 新洞察生成
run.completed               — 试映运行完成

audience.generation.job.started   — 观众生成后台任务开始
audience.generation.job.completed — 观众生成后台任务完成
audience.generation.job.failed    — 观众生成后台任务失败
audience.generation.job.canceled  — 观众生成后台任务取消
audience.plan.started             — 采样计划生成开始
audience.plan.progress            — real provider 采样计划流式进度
audience.plan.ready               — 采样计划可审核
audience.plan.updated             — plan 或 directive 被编辑
audience.plan.confirmed           — 用户确认采样计划
audience.plan.failed              — 采样计划生成失败
audience.profile.expansion.started — directive profile 扩展开始
audience.profile.expansion.ready   — directive profile 扩展完成
audience.profile.expansion.directive_failed  — directive profile 扩展失败
audience.identity.started          — 单个 profile 人设开始生成
audience.identity.ready            — 单个 profile 人设生成完成
audience.identity.failed           — 单个 profile 人设生成失败
audience.updated                  — profile 或人设信息更新
run.pausing                       — 试映正在暂停

run.resumed                       — 试映已继续
run_log.created                   — 运行日志创建
```

> **完整清单**：以上为运维排障常用事件。完整 SSE 事件类型清单（含 `comment.updated`、`audience.plan.frame`、`audience.profile.created`、`run.clock.updated`、`run.started` 等）见 `02_API契约与共享DTO.md` 第 8 节。

### 13.9 行为过滤规则

并非所有 tool call 都会触发 LiveUpdate。过滤规则：

```text
触发动画/UI 的关键行为（仅这些触发 audience.action_happened）：
  open_post, like_post, favorite_post, share_post, write_comment, like_comment, exit_browsing

仅记录到 agent_tool_calls、不触发 LiveUpdate 的内部浏览行为：
  普通停留, 普通阅读, 仅心路历程轮次
```

排障时如果观众在行动但 UI 无反应，首先确认该行为是否属于上述关键行为之一。

### 13.10 animationHint 字段

`audience.action_happened` 事件携带 `animationHint` 字段，用于驱动前端动画，排障动画问题时需检查此字段：

```text
animationHint 取值：
  "heart"   — 点赞动画
  "star"    — 收藏动画
  "comment" — 评论动画
  "risk"    — 质疑/风险动画
  "skip"    — 跳过动画（配合 exit_browsing action 使用）
  "none"    — 无动画
```

注意："跳过"在 UI 层显示为已跳过；事件层用 `exit_browsing` + `animationHint: "skip"` 表达，不新增 `skip_post` 工具。

### 13.11 LLM 调用与 token 消耗追踪

LLM 调用追踪分两层：

```text
llm_call_traces          每次 provider call 的明细事实
run_llm_usage_summaries  run 级累计 token，用于审计和后续实时显示
```

记录来源是 AI SDK telemetry。`recordInputs / recordOutputs` 必须保持关闭；完整请求/响应回放使用 `agent_turns.request_json / raw_response_json`，token 追踪表只保存 usage 和非敏感元数据。

AI SDK tool loop 的 multi-step 不是重复统计错误。每个 step 都会向 provider 发起一次请求，并重新发送必要上下文，所以成本审计必须逐 step 累加。

排查 SQL：

```sql
-- run 总 token 消耗
SELECT *
FROM run_llm_usage_summaries
WHERE run_id = 'run_001';

-- 按任务类型拆分 token 消耗
SELECT
  task_type,
  COUNT(*) AS call_count,
  SUM(input_tokens) AS input_tokens,
  SUM(output_tokens) AS output_tokens,
  SUM(total_tokens) AS total_tokens
FROM llm_call_traces
WHERE run_id = 'run_001'
GROUP BY task_type
ORDER BY total_tokens DESC;

-- 查看单个 AgentTurn 内每个 AI SDK step 的真实消耗
SELECT
  step_number,
  model,
  finish_reason,
  input_tokens,
  output_tokens,
  total_tokens,
  metadata_json
FROM llm_call_traces
WHERE agent_turn_id = 'turn_001'
ORDER BY step_number, created_at;
```

### 13.12 基础设施日志

除上述领域级事件追踪外，系统还输出结构化基础设施日志用于服务端排障。

#### 日志库与格式

```text
日志库：pino（Fastify 内置）
输出格式：JSON（生产环境）/ pino-pretty（本地开发）
配置入口：apps/api/src/logger.ts
日志级别：通过 LOG_LEVEL 环境变量控制（默认 info，可选 debug/warn/error）
```

所有模块通过 `import { log } from "../logger.js"` 使用统一 logger，不再直接使用 `console.*`。

#### 请求日志

每个 HTTP 请求完成后自动输出：

```json
{
  "level": 30,
  "method": "POST",
  "url": "/api/runs/run_001/start",
  "statusCode": 200,
  "elapsed": 42,
  "msg": "request completed"
}
```

- 2xx 请求：info 级别
- 4xx 请求：warn 级别
- 5xx 请求：error 级别

#### API 错误日志

所有通过 `sendApiError` 返回的错误都会记录：

```json
{
  "level": 40,
  "statusCode": 409,
  "code": "INVALID_RUN_STATUS",
  "message": "只有运行中的试映才能暂停",
  "msg": "API error: INVALID_RUN_STATUS"
}
```

- 4xx 业务错误：warn 级别
- 5xx 服务器错误：error 级别（含完整 error 对象）

#### Scheduler 日志

关键生命周期事件：

```text
[info]  run loop started          — run 调度循环启动
[info]  run loop ended            — run 调度循环结束
[error] run loop crashed          — 调度循环异常退出（含 error 对象）
[warn]  interrupted journey       — 启动时发现中断的 journey runner
[info]  recovering active runs    — 启动时恢复活跃 run
[error] journey failure handler   — 处理 journey 失败时的二次异常
[warn]  emitRunSummary failed     — 非关键的摘要推送失败
```

#### 报告恢复日志

启动时自动恢复中断的报告生成任务：

```text
[info]  recovering stale report_generating runs — 发现待恢复的报告任务
[error] failed to recover report generation     — 恢复失败（含 runId 和 error）
```

#### Prisma 慢查询日志

数据库查询超过 200ms 自动告警：

```json
{
  "level": "warn",
  "module": "prisma",
  "query": "SELECT \"public\".\"test_runs\"...\"",
  "duration": 342,
  "target": "quaint::connector::query"
}
```

#### 基础设施日志排障示例

查看某 run 的所有基础设施日志：


排查 Scheduler 卡住：

### 13.13 常见问题排查

#### 前端没有实时刷新

检查：

```text
1. SSE 是否连接成功。
2. 是否收到 heartbeat。
3. live_update 是否写入 / 推送。
4. eventId 是否被前端误判重复。
5. 是否仍在使用 V1 唯一实时通道 `/api/runs/:runId/events`。
```

#### 点赞数重复增加

检查：

```text
1. same journey 是否重复 like_post。
2. 幂等约束是否生效。
3. action retry 是否重复提交工具事务。
4. agent_tool_calls 是否重复 committed。
```

#### 评论出现但 comment_count 没变

检查：

```text
1. write_comment 是否 committed。
2. simulated_comments 是否写入。
3. simulated_post_states.comment_count 是否同事务更新。
4. post_state.updated 是否推送。
```

#### Agent 总像分析师

检查：

```text
1. Audience Agent system prompt 是否强调普通用户。
2. thought_text 示例是否太像分析报告。
3. persona 是否过于理性。
4. 是否缺少小红书评论语气示例。
```

#### Agent 工具乱用

检查：

```text
1. 当前 session 的工具集是否正确（feed tools vs post tools）。
2. Prompt 是否按 session 阶段固定工具协议。
3. 工具服务是否正确返回 ok: false / reason 对非法调用。
4. Repair 是否触发过多。
```

#### Session 阶段切换异常

检查：

```text
1. open_post 是否正确完成 feed session 并创建 post session。
2. post session 的 initial observation 是否包含完整帖子详情。
3. 是否存在 feed session 中泄漏 post tools 的情况。
4. 恢复时 feed session completed 但 post session 缺失，是否从 DB contentVersion 正确重建。
```

#### real AI run 配置或调用失败

检查：

```text
1. config/llm.local.yaml 中 runtimeMode 是否为 real。
2. real 模式下，apiKey 是否存在且只在服务端读取。
3. baseUrl 是否符合 OpenAI-compatible 服务要求。
4. real 模式下，models.fast / models.pro 是否已配置，且 models.pro 是否支持图片输入。
5. 模型是否支持 vision，能处理封面图片。
6. 本地测试是否应切换 runtimeMode=mock。
```

#### 自动轮转后系统变慢

检查：

```text
1. 是否错误使用 sleep。
2. Scheduler 是否把默认 delay 当成真实等待。
3. Scheduler 是否按 queue_seq 排序快速执行。
```

#### run 一直不完成

检查：

```text
1. 是否存在 active journeys 但无 pending actions。
2. 是否存在 running action 超时未恢复。
3. 是否自动轮转逻辑没有生成下一条 action。
4. 是否 journey 超过最大 step 但未终止。
```

#### 工具调用重复执行（幂等调试）

检查：

```text
1. 查 agent_tool_calls 表中同一 journey、同一 tool_name、同一 input 是否出现多条 committed 记录。
2. 检查 tool_call_id 是否重复：同一 tool_call_id 是否被 commit 了多次。
3. 检查 action retry 时是否跳过了已 committed 的 tool call。
4. Direct Tool Commit 模式下，检查 commit 阶段是否对 (journey_id, tool_call_id) 做了幂等去重。
```

排查 SQL：

```sql
-- 查找重复执行的 tool call
SELECT tool_name, input, COUNT(*) AS commit_count
FROM agent_tool_calls
WHERE journey_id = 'journey_001'
  AND status = 'committed'
GROUP BY tool_name, input
HAVING COUNT(*) > 1;

-- 查找同一 tool_call_id 是否被多次 commit
SELECT tool_call_id, COUNT(*) AS cnt
FROM agent_tool_calls
WHERE journey_id = 'journey_001'
GROUP BY tool_call_id
HAVING COUNT(*) > 1;
```

#### SSE 事件未推送 / 前端漏收

检查：

```text
1. live_events 表是否有对应事件记录，确认事件已写入。
2. SSE 连接是否在事件产生时仍存活（检查 last_heartbeat_time）。
3. 服务端是否在工具事务提交后调用 pushSSE。
4. 前端 EventSource 是否因网络抖动断开后未重连。
5. SSE id 是否使用 live_events.sequence 字符串。
6. Last-Event-ID 回放是否按 sequence 查询。
```

排查 SQL：

```sql
-- 查最近事件是否写入，并确认 sequence 可用于 SSE id
SELECT sequence, event_type, payload, created_at
FROM live_events
WHERE run_id = 'run_001'
ORDER BY sequence DESC
LIMIT 20;

-- 查某时间段内所有事件
SELECT sequence, event_type, payload, created_at
FROM live_events
WHERE run_id = 'run_001'
  AND created_at BETWEEN '2026-01-01' AND '2026-01-02'
ORDER BY sequence;
```

#### 使用 live_events 回放现场

当需要还原某个时间点的直播现场状态时：

```text
1. 从 live_events 表筛选目标 run_id 的所有事件。
2. 按 sequence 排序。
3. 逐条重放：还原评论区、点赞数、帖子状态等。
4. 结合 agent_turns 和 agent_tool_calls 交叉验证每条事件的触发来源。
```

回放 SQL：

```sql
-- 回放某 run 的完整现场事件流
SELECT sequence, event_type, payload, created_at
FROM live_events
WHERE run_id = 'run_001'
ORDER BY sequence;

-- 回放仅评论区事件
SELECT sequence, event_type, payload, created_at
FROM live_events
WHERE run_id = 'run_001'
  AND event_type IN ('comments.page_loaded', 'comment.created')
ORDER BY sequence;
```

#### 追溯评论来源（write_comment 追踪）

当需要确认某条评论来自哪次 `write_comment` 调用时：

```text
1. 通过 simulated_comments.tool_call_id 关联 agent_tool_calls。
2. 查看该 tool call 的 input 获取评论原文和回复对象。
3. 回溯到 agent_turn 获取当时的上下文（agent_turn_contexts）。
4. 确认该 comment 是否随 Direct Commit 正确写入并推送。
```

排查 SQL：

```sql
-- 完整追溯：从评论文本到 write_comment tool call 到 turn 上下文
SELECT
  c.id AS comment_id,
  c.comment_text,
  c.parent_comment_id,
  tc.id AS tool_call_id,
  tc.input,
  tc.output,
  t.id AS turn_id,
  t.step_index,
  tc2.input_context_json
FROM simulated_comments c
JOIN agent_tool_calls tc ON c.tool_call_id = tc.id
JOIN agent_turns t ON tc.agent_turn_id = t.id
LEFT JOIN agent_turn_contexts tc2 ON tc2.agent_turn_id = t.id
WHERE c.id = 'comment_001';
```

#### 查看 Agent 对封面图的反应

当需要查看 Agent 视觉模型对封面图的处理结果时：

```text
1. 查 agent_turn_contexts.input_context_json，确认 feed initial_observation 的 multimodal message 已包含 image_url。
2. 查 agent_turns 对应的 agent_transcript_items，确认是否包含基于封面图的第一眼反应。
3. 检查 multimodal 消息中 image_url 是否正确传入模型。
4. 对比 thought_text 与封面内容，判断视觉理解是否准确。
```

排查 SQL：

```sql
-- 查某 journey 中所有 feed 阶段的 turn 上下文
SELECT
  t.id AS turn_id,
  t.step_index,
  tc.input_context_json
FROM agent_turns t
LEFT JOIN agent_turn_contexts tc ON tc.agent_turn_id = t.id
WHERE t.journey_id = 'journey_001'
ORDER BY t.step_index;
```

#### 模型无法读取图片

当 Agent 报告"看不到图片"或图片理解异常时：

```text
1. 检查图片 URL 是否可访问（是否过期、是否需要鉴权）。
2. 确认 multimodal message 格式是否正确：image_url 需为 {"type": "image_url", "image_url": {"url": "..."}} 结构。
3. 检查模型是否支持 vision，非 vision 模型会忽略图片。
4. 查看 SDK 返回的 usage 中是否有 image_tokens > 0，若为 0 说明图片未被处理。
5. 检查图片大小是否超过模型限制（通常 20MB base64 / 限制分辨率）。
6. 查看 raw_agent_output_json 或 Agent 错误日志中是否有模型关于图片的错误反馈。
```

排查 SQL：

```sql
-- 查某 turn 的模型输入中是否包含图片
SELECT
  t.id AS turn_id,
  tc.input_context_json,
  t.status
FROM agent_turns t
LEFT JOIN agent_turn_contexts tc ON tc.agent_turn_id = t.id
WHERE t.id = 'turn_001';
```

#### 点击发布后页面未跳转但 run 未启动

检查：

```text
1. 前端是否正确调用发布接口，检查 Network 请求是否发出及返回状态码。
2. 后端发布接口是否返回错误（检查 response body 中的 error message）。
3. run 记录是否仍为 draft、audience_ready、running 或 paused。
4. Scheduler 是否正常运行，检查 Scheduler 服务日志是否有异常。
5. 是否存在参数校验失败（如 content_version 未就绪、audience 配置缺失等）导致 run 创建被拒绝。
6. 前端路由跳转逻辑是否依赖 run.started 事件，若 run 未启动则跳转不会触发。
```

排查 SQL：

```sql
-- 查某次发布的 run 状态
SELECT id, status, created_at, started_at, error_message
FROM test_runs
WHERE id = 'run_001'
ORDER BY created_at DESC
LIMIT 5;

-- 查 run 是否有对应的 run participant 记录
SELECT *
FROM run_participants
WHERE run_id = 'run_001';
```

#### SSE 已连接但日志不刷新

检查：

```text
1. 确认 SSE 连接已建立：浏览器 DevTools > EventSource 面板中连接状态为 open。
2. 检查 heartbeat 是否正常收到：若无 heartbeat，说明服务端推送通道异常。
3. 检查前端是否连接 `/api/runs/:runId/events`。
4. 确认 run 是否仍在运行：若 run 已完成或暂停，不会有新的日志事件产生。
5. 检查后端 action 是否卡在某个状态（如 running 但长时间无 tool call 产出）。
6. 查看 live_events 表是否有新事件写入，并确认 sequence 连续递增。
7. 检查是否存在前端事件处理逻辑异常（如 onmessage 回调报错导致后续事件被忽略）。
```

排查 SQL：

```sql
-- 查最近的 live_events 是否正常写入
SELECT sequence, event_type, payload, created_at
FROM live_events
WHERE run_id = 'run_001'
ORDER BY sequence DESC
LIMIT 20;

-- 查当前活跃 turn 的状态
SELECT id, status, started_at, updated_at
FROM agent_turns
WHERE journey_id IN (SELECT id FROM agent_journeys WHERE run_id = 'run_001')
  AND status IN ('model_calling', 'tools_executing')
ORDER BY updated_at DESC;
```

#### 左侧内容锁定失败

检查：

```text
1. 确认前端 workspaceStatus 是否已进入 starting / running / paused / report_generating / completed。
2. 确认表单 disabled / readOnly 逻辑是否绑定到 workspaceStatus，而不是依赖 content_versions.status。
3. 确认 POST /api/runs 已成功返回 runId，且当前页面保存了发布时的内容快照。
4. 确认 run 启动或系统异常暂停时，前端是否仍保留已创建 run 的内容快照和错误原因。
5. 注意：V1 的 content_versions 表没有 locked/status 字段，内容锁定是前端 UI 只读状态和已创建 content_version 快照共同保证。
```

排查 SQL：

```sql
-- 查发布时保存的 content_version 快照
SELECT id, run_id, version_name, title, cover_image_url, scale, created_at
FROM content_versions
WHERE run_id = 'run_001';
```

#### Agent 引用了不存在的目标人群

检查：

```text
1. 查看 agent 的 system prompt 中是否硬编码了目标人群描述，而非从配置动态获取。
2. 检查 profile_snapshot_json / agent_snapshot_json 中的 participant 快照是否与实际 run_participants 记录一致。
3. 确认 prompt 模板中的 audience 变量是否正确替换（检查渲染后的 prompt 原文）。
4. 查看 run_participants 表中是否存在该 participant_id 的记录。
5. 检查是否是 prompt 版本不匹配：prompt 引用了当前采样计划之外的人群定义。
6. 确认 run 创建时 audience 配置是否完整传入（检查 run 的 audience_config 字段）。
```

### 13.18 已知风险

#### Scheduler 全局并发配置

`Scheduler.resolveConcurrency()` 的并发数由全局 `SCHEDULER_DEFAULT_CONCURRENCY` 决定，不再读取 run 级并发快照，也不再维护独立 admission window。启动日志会打印当前生效的 scheduler/LLM 配置。

**风险**：如果 `SCHEDULER_DEFAULT_CONCURRENCY` 环境变量设为过大值（如容量测试后忘记改回），会同时启动大量 agent runner，导致：
- 数据库连接池耗尽
- LLM API 排队时间变长；真实 HTTP 请求由 `LlmCapacityManager` 的 RPM + 并发双限流兜底（见 `03_Agent运行时设计.md` 9.3 节）
- 内存压力增大

**建议**：本地真实模型试映将 `SCHEDULER_DEFAULT_CONCURRENCY` 设为 1–3。LLM 容量配置通过 `config/llm.local.yaml` 的 `capacity` 字段管理，支持热重载；修改 `.env.local` 后需要重启 API 进程生效，但修改 `capacity` 字段可通过设置页保存即时生效。

#### LLM 容量校准费用风险

`POST /api/settings/llm/capacity/probe` 会向当前模型发送真实请求（每档并发测试 60 秒，档间冷却 60 秒，使用 prompt `"不思考，回复1"` + `max_tokens: 1`），可能产生 API 费用。校准是 job-based 异步任务，可能运行数分钟。校准结果受当前网络、账号、模型和时间波动影响。用户必须在前端确认费用提示后才会执行校准；校准结果不会自动写入配置，需要用户点击"应用推荐值"才会持久化。

校准 prompt 选择 `"不思考，回复1"` + `max_tokens: 1` 的设计原因：用最短 prompt 和最短输出降低单次请求成本，避免校准本身消耗过多 token；同时让请求主要受网络和限流影响，而非受模型思考时间影响，从而测出真实容量上限。每档 60 秒窗口足以观察 RPM 稳定值，档间 60 秒冷却避免不同档位请求打在同一个限流窗口。

#### LLM 容量配置排障

```text
症状：真实模型请求持续 429 / 503
排查：
  1. GET /api/settings/llm/capacity/status 查看 cooldownUntil / lastLimitReason
  2. 如果 effectiveRpm 已降到 minRpm，说明 AIMD 已触发降速；检查 provider 侧额度
  3. 尝试切换 preset 到 conservative（initialRpm=4, minRpm=1, maxRpm=30, initialConcurrency=2, minConcurrency=1, maxConcurrency=2）
  4. 必要时点击"重置自动学习结果"恢复 initial 值

症状：请求排队时间过长
排查：
  1. GET /api/settings/llm/capacity/status 查看 queueSize / inFlight
  2. 如果 queueSize 持续增长，说明 effectiveConcurrency 过低或 provider 响应慢
  3. 在 auto 模式下等待连续成功后自动上调；或切到 manual 模式手动提高 maxConcurrency
  4. 高额度账号可切到 high_quota 预设（initialRpm=30, minRpm=5, maxRpm=300, initialConcurrency=8, minConcurrency=1, maxConcurrency=16）
```
