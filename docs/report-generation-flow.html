两条生成轨道

real 轨道依赖模型结构化输出；fallback 轨道是关键词和计数规则。

Real LLM 轨道

配置门槛`runtimeMode=real` 且 baseUrl、apiKey、audience model、report model 完整。
输入内容标题、正文、postState、journey 概况、观众列表、action logs 文本、comments 文本、图片 URL。
模型要求系统 prompt 要求只输出 JSON，包含 recommendation、summary、dimensions、commentPreview、risks、revisionSuggestions、evidenceIndex。
失败处理SDK 抛错时暂停 run，写 errorMessage，推送 `run.paused`，不会降级成 mock 报告。

Mock / fallback 轨道

推荐规则打开人数不足半数则不推荐；否则有信任质疑则修改后发布；否则推荐发布。
维度分析根据打开、收藏、评论计数和关键词生成 firstImpression、trust、value、interactionIntent、risks。
评论预演按“吗 / 求 / 价格 / 广告 / 码住”等关键词把评论分到 question、doubt、resonance。
证据索引简单把前 20 条 log 和 comment 的文本截断成 claim，再挂原始 id。

当前问题清单

按对报告可信度和用户决策风险排序。

Real 输出没有 schema 校验

P0

模型输出只做 JSON.parse，解析失败会返回空对象；字段类型错误也会被直接写库。

证据：`reportAgent.ts:107-123` 捕获 parse 失败后 `parsed = {}`，然后给空 sections 默认值。
修复方向用 shared DTO / zod 校验报告结构；real 模式结构不合格应进入 REPORT_GENERATION_FAILED，不应写成功报告。

证据输入少于文档要求

P0

文档要求 tool calls、social events、social reactions、insights、source breakdown；当前报告只取了部分数据。

证据：`report.ts:36-39` 只查询 journeys / comments / actionLogs / audiences；`docs/08_Report_Generation_Spec.md:29-51` 要求更多输入。
修复方向建立 report evidence builder，把所有报告证据统一抽成结构化输入，再同时供 real 和 fallback 使用。

证据索引不可真正验证

P1

fallback 的 evidenceIndex 是文本截断；real prompt 只给日志和评论文本，没有明确 id 列表让模型引用。

证据：`buildEvidenceIndex()` 用 `slice(0, 40)` 生成 claim；`reportAgent.ts:61-79` prompt 拼的是纯文本。
修复方向给模型输入 `{id,type,text,participant,source,simulatedTime}`；输出只能引用已有 id，落库前校验 evidenceRefs 存在。

报告生成超时没有显式控制

P1

文档写 Report 生成超时 180 秒，但当前 LLM 调用没有独立 AbortController 或 timeout 参数。

证据：`docs/04_Runtime_Scheduler_Runner_Design.md:603-608` 定义 180 秒；`reportAgent.ts:99-106` 直接等待 SDK。
修复方向给报告模型调用加 180 秒超时；超时后走统一 REPORT_GENERATION_FAILED 暂停流程。

前端不展示 evidenceIndex

P1

API 返回 evidenceIndex，但前端类型和 ReportPanel 都没有展示；用户看不到“结论来自哪些行为”。

证据：`views.ts:91` 返回 evidenceIndex；`apps/web/src/types.ts:141-150` 没有 evidenceIndex 字段；`ReportPanel.tsx` 不渲染证据索引。
修复方向在报告页加证据链区域；支持按结论展开相关 action/comment/tool call。

fallback 文案会放大证据

P2

只要有一条信任质疑评论，就可能输出“多位观众追问”；这会让报告显得比事实更严重。

证据：`buildFallbackReport()` 对 `trustComments.length` 只判断是否大于 0。
修复方向把数量、比例、来源拆分写进文案生成规则；避免“多位”“集中”等没有阈值支撑的措辞。
