# 07 测试用例与验收文档

## 1. 目标

本文档定义 AI 用户试映会 V1 的测试用例、验收标准和 Prompt 评测方法。

验收目标：

```text
1. 系统能跑通单版本 AI 试映现场。
2. Agent 通过 `thought_text` 记录心路历程，通过 tool use 完成外部行为。
3. 多 tool call 回合能正确落库、校验、提交和展示。
4. 点赞、收藏、评论能真实改变模拟帖子状态。
5. 失败可重试，中断可恢复。
6. 报告基于证据生成，而不是凭空分析。
```

CI 分层：

```text
PR CI 运行 smoke integration，覆盖 plan confirm、重复 confirm、缩小规模 mock run、open/skip 行为和核心 tool executor 事件；smoke 可在测试配置中降低单个用例的 Scheduler 并发，以适配 SQLite runner 的写入能力。
完整 integration 保留在本地 pnpm verify、手动或后续 nightly workflow 中运行。
SQLite CI 不通过降低 profile expansion、identity generation 或 AgentRunner 并发来换取稳定性；需要远端完整高并发验证时，应使用更接近生产并发特性的数据库服务。
```

前置：

```text

```

预期：

```text

5. 缺少 runtimeMode 的请求或 YAML 配置不兼容，返回校验错误或启动失败。
```

## 5. 观众生成阶段测试

### TC-100 观众采样计划生成

前置：

```text
run 创建成功
```

操作：

```text
调用 POST /api/runs/:runId/audience-sampling-plan
```

预期：

```text
1. 接口快速返回 active job 和 plan snapshot，生产环境和测试环境都不等待完整身份生成。
2. run.status 进入 planning_audience。
3. 后端任务推送 audience.generation.job.started / audience.plan.started。
4. 计划生成完成后推送 audience.plan.ready；GET /audience-sampling-plan 返回 planMarkdown、dimensions 和 directives。
5. 此时 audience_profiles count = 0，用户审核对象不是单个 profile。
6. sum(directive.quantity) = plan.totalCount。
```

### TC-101 采样指令编辑与校验

前置：

```text
AudienceSamplingPlan.status = ready_for_review
```

操作：

```text
1. PATCH /api/runs/:runId/audience-sampling-plan 更新 planMarkdown 或 dimensions；totalCount 由 directive quantity 合计同步。
2. 用户点击单组编辑按钮，PATCH /api/runs/:runId/audience-sampling-plan/directives/:directiveId 修改 quantity 或 description。
3. 用户点击单组删除按钮，DELETE /api/runs/:runId/audience-sampling-plan/directives/:directiveId 删除 directive。
4. 用户通过"优化观众分布"建议卡片应用新增操作时，前端调用 POST /api/runs/:runId/audience-sampling-plan/directives 新增 directive。
```

预期：

```text
1. 未确认 plan 可以编辑；确认后所有 directive 结构写操作被拒绝。
2. 修改 directive 后同步 plan.totalCount 和 TestRun.audienceCount，不改写 planMarkdown。
3. 前后端都校验 quantity 总和；确认时总和不等于 totalCount 必须失败。
4. 推送 audience.plan.updated。
```

### TC-101A 优化观众分布对话与建议卡片

前置：

```text
AudienceSamplingPlan.status = ready_for_review
当前 plan 至少包含 2 条 directive
```

操作：

```text
1. 打开"优化观众分布"弹窗。
2. 在输入框输入 @，前端弹出当前所有分组，并支持按 name / description / diversityAxes 模糊搜索。
3. 选择"核心用户"后，输入"把 @核心用户 里的预算敏感人群拆出来"。
4. 前端显示消息为 @核心用户 chip，但发送给 revision-suggestions 的该消息末尾包含被 @ 分组的完整结构化上下文。
5. 后端按 directiveId 读取当前 DB 中最新 directive，调用 revision agent，返回 PlanRevisionProposal。
6. 前端把 proposal 渲染为建议卡片，展示新增 / 修改 / 删除 operation，以及修改字段的 before / after。
7. 用户点击"应用到采样计划"，前端按 operation 顺序调用现有 directive CRUD API，不调用 apply 或 batch API。
```

预期：

```text
1. 只手动输入 @核心用户 而未从列表选择时，不作为 mention，不附带隐藏上下文。
2. revision-suggestions 不修改数据库，不推送 audience.plan.updated。
3. 应用建议时每条 operation 都走现有后端校验；directiveId 不属于当前 plan 时失败。
4. 应用成功的 operation 在建议卡片中标记 applied，并用最新 plan snapshot 更新页面。
5. 应用失败的 operation 在建议卡片中标记 failed，显示后端错误；已成功和失败的 operation 必须可区分。
6. 删除到 0 个 directive 时，前端禁止应用，后端也返回错误。
7. confirmed plan、active audience generation job、running 之后不能打开或提交优化建议。
```

### TC-102 确认采样计划并自动扩展

前置：

```text
AudienceSamplingPlan.status = ready_for_review
且 directive 数量总和等于 totalCount
```

操作：

```text
调用 POST /api/runs/:runId/audience-sampling-plan/confirm
```

预期：

```text
1. plan.status 进入 confirmed / expanding_profiles，confirmedAt 写入。
2. 推送 audience.plan.confirmed 和 audience.profile.expansion.started。
3. 每个 directive 按 chunk 流式创建对应数量 AudienceProfile；每创建一个合法 profile 推送 audience.profile.created，profile.samplingDirectiveId 和 sampleIndex 正确写入。
4. 扩展完成的 directive 推送 audience.profile.expansion.directive_ready；失败 directive 推送 audience.profile.expansion.directive_failed 并保留重试入口。
5. profile 创建为 profile_only 后，identity pool 不等待全部 expansion 完成即可 claim 并生成人设，不需要用户额外触发身份生成。
6. Profile Expander provider 输入不包含 existingProfiles；每个 directive 只接收 confirmed plan、当前 directive 和内容事实。
7. 当存在多个 directive 时，profile expansion 允许多个 directive chunk 并发执行；默认最多 3 个真实 LLM 调用。任一 directive 重试前必须清理该 directive 下已有 profiles / identities；正常生成中 profile 可以逐个写库并推送。
8. 并发提交两个 confirm 请求时，只允许一个返回 200 并创建一条 active audience generation job；另一个返回 409，不得返回 500 或创建重复 job。
```

### TC-103 观众身份生成与恢复补偿

前置：

```text
存在 active audience_generation_jobs，scope 为 profile_expansion / identities，且部分 profile 处于 identity_queued / identity_generating
```

预期：

```text
1. 每个 profile 先进入 identity_queued / identity_generating，再逐个落库 identity_ready 或 identity_failed。
2. 身份生成推送 audience.identity.started / audience.identity.ready / audience.identity.failed。
3. 全部可生成项处理完成后 job.status 变为 completed；若所有 profile 都 identity_ready，run.status 变为 audience_ready。
4. worker 在取消、inactive 或锁过期后返回，不能把 job 覆盖为 completed，也不能发送 audience.generation.job.completed。
5. recoverAudienceGenerationJobs 会处理 stale active job：可恢复 job 重新领取执行；不可恢复或超过重试上限的 job 标 failed 并释放 profile。
6. 一致性扫描会修复 identity_queued / identity_generating 但 job 不存在、inactive 或已终止的孤儿 profile。
7. job 失败或取消后，run 不得卡在 planning_audience；根据当前 plan/profile/job 状态恢复为 draft / generating_audience / audience_ready。
8. pipeline 内 identity pool 使用数据库状态原子 claim profile_only / identity_failed profile；默认最多 10 个 profile 同时生成；每个 profile 仍单独调用 Persona Generator、单独事务提交，Persona Generator 输入不包含 existingProfiles / existingPersonas。
9. 并发生成中单个 profile 失败只影响该 profile 的 identity_failed 和 directive 进度，不阻断同 batch 其他 profile 完成。
10. real provider 的非流式 chat completion 调用遇到 429 / rate limit、408、5xx 或网络超时会先做短退避重试；重试耗尽后才进入原有 directive/profile 失败路径。采样计划 streaming 生成和报告生成不在该规则覆盖范围内。结构化输出校验失败不做退避重试。
```

```text
[ ] 顶部 HUD 包含控制按钮（暂停/继续/结束并生成报告）
[ ] 右侧运行席内实时日志展示生成、调度、思考、动作、结果和异常
[ ] 右侧运行席内实时日志可筛选，支持滚动自动加载更多；日志展开时观众席收起为摘要栏
[ ] 暂停时显示待执行步数和控制选项
```

结果层：

```text
[ ] 报告回答能不能发
[ ] 说明为什么点开
[ ] 说明为什么直接结束浏览
[ ] 说明为什么收藏
[ ] 说明为什么质疑
[ ] 展示评论区预演
[ ] 给出具体修改建议
[ ] 证据可追溯
```

## 15. Prompt 评测

系统跑通不等于产品成立。真正要评估的是：

```text
Agent 是否像真实小红书用户？
行为链是否自然？
评论是否真实？
反应是否有差异？
报告是否基于证据？
```

观众生成评测覆盖 Sampling Plan Generator、Sampling Plan Revision Agent、Profile Expander、Persona Generator 和 Seat Revision Agent。采样计划结构、directive 展开数量校验和 mock/real provider 输出约束见 `04_观众生成领域规格.md`。

### 15.1 评测对象

```text
Audience Sampling Plan Generator Prompt
Audience Sampling Plan Revision Prompt
Audience Profile Expander Prompt
Audience Persona Generator Prompt
Audience Seat Revision Prompt
Audience Agent Feed Session Prompt
Audience Agent Post Session Prompt
Repair Prompt
Report Generator Prompt
```

### 15.2 评测方式

采用三层评测：

```text
1. 自动规则检查
2. 人工抽检
3. Demo 对比评测
```

### 15.3 自动规则检查

#### 15.3.1 Tool 合法性

指标：

```text
非法 tool call 数量
ignored tool call 占比
无 tool call action 数量
content 为空且无 tool_calls 的无效 action 数量
content 为空但有 tool_calls 的工具回合数量
repair 触发次数
```

阈值建议：

```text
ignored tool call 占比 < 10%
content 为空且无 tool_calls 的无效 action = 0
repair 触发率 < 15%
```

#### 15.3.2 会话阶段与工具合法性

检查：

```text
feed_card 阶段是否出现 post tools（like_post、write_comment 等）
post_detail 阶段是否出现 feed tools（open_post）
open_post 后是否正确切换到 post_detail
exit_browsing 后是否继续状态变更
comments_view 翻页 cursor 是否有效
```

#### 15.3.3 评论结构完整性

检查：

```text
content 非空
content 是否符合 persona 语气
reply_to_comment_id 为空时是否回复主帖
reply_to_comment_id 有值时是否属于当前帖子评论
是否避免违法、色情、仇恨、真实人身攻击
```

#### 15.3.4 采样计划说明可读性

`planMarkdown` 除了内容相关性，还需要检查移动端扫读结构：

```text
总长度 160-280 个中文字符
使用 2-3 个以空行分隔的自然段落
每段集中表达一个完整意思，避免单个长段落承载全部信息
列表和加粗可以使用，但不作为必选结构
不使用复杂标题层级、表格、长列表或固定模板
```

Prompt eval 脚本和真实模型评测结果不进入 Git 仓库；如需重复调用真实配置评测，应在本地私有脚本中读取 `LLM_CONFIG_PATH`，避免打印 API key，并只输出必要的脱敏摘要、结构化结果和可读性指标。

#### 15.3.5 心路历程完整性

检查：

```text
content 非空时，是否能作为 thought_text 保存
thought_text 是否像用户主观反应，而不是分析报告
thought_text 是否自然解释本轮行为
thought_text 是否避免打分和营销术语
content 为空但有 tool_calls 时，系统是否没有伪造 fallback thought_text
```

### 15.4 人工抽检维度

#### 15.4.1 角色感

问题：

```text
这个 Agent 像真实用户，还是像内容分析师？
```

好例：

```text
"这个标题挺戳我，但我怕又是广，先点进去看看有没有具体型号。"
```

坏例：

```text
"该标题通过痛点刺激提升用户点击意愿，但信任背书不足。"
```

评分：

```text
1 = 完全像分析师
3 = 偶尔像用户
5 = 非常像真实用户
```

#### 15.4.2 行为自然度

检查：

```text
是否先看再点
是否会在没兴趣时直接结束浏览
是否会只有想法但暂时无外部动作
是否会看评论区
是否会收藏但不点赞
是否会评论但不一定正向
```

坏例：

```text
每个用户都按 open_post → like_post → favorite_post → write_comment → exit_browsing 固定流程走。
```

#### 15.4.3 人设差异

检查：

```text
核心用户 vs 路人用户反馈是否不同
专业用户是否会纠错
广告敏感用户是否会质疑
低注意力用户是否会直接结束浏览
Agent 是否能在没有目标人群输入的情况下自动推断受众
Agent 是否能基于标题、封面图、正文生成合理 persona
Agent 是否没有引用不存在的账号背景 / 目标人群 / 观察问题
```

坏例：

```text
30 个 persona 只是名字不同，反馈模式完全一样。
```

#### 15.4.4 评论真实感

好评论：

```text
"蹲一个具体型号，别又是广吧"
"这个坑我真踩过，安装后才知道麻烦"
"说得有点绝对，不同户型差挺大的"
```

坏评论：

```text
"我认为本文的信息密度较高，建议作者补充数据来源。"
```

#### 15.4.5 质疑自然度

检查：

```text
是否允许质疑
质疑是否符合人设
质疑是否有具体原因
是否避免无意义攻击
```

#### 15.4.6 心路历程质量

好的 thought_text：

```text
像真实用户当下反应，而不是复盘报告。
能解释为什么点开、收藏、评论、看评论或结束浏览。
能表达不确定、犹豫、质疑、懒得看等状态。
```

坏的 thought_text：

```text
"该内容具备较强转化潜力。"
只输出工具调用，没有任何用户心路历程。
用 attention/trust/value 等评分语言评价自己。
```

#### 15.4.7 报告证据链

检查：

```text
报告结论是否能追溯到评论、日志、行为。
修改建议是否有证据。
是否出现无证据泛泛判断。
```

#### 15.4.8 多模态评估（封面图 + 视觉能力）

前提：系统使用真实封面图 + 多模态 AI 视觉能力进行评测。

检查：

```text
Agent 是否真的基于封面图给出反馈，还是只靠文字描述？
Agent 是否幻觉出图中不存在的内容？
Agent 能否区分「标题吸引力」和「封面吸引力」？
当图片信息不足时，Agent 能否表达"不确定"？
```

坏例：

```text
封面是一张模糊的室内照片，Agent 却说"这个装修风格很北欧简约，很吸引我"。
实际封面只有产品实物图，Agent 却说"封面上的模特穿搭很好看"。
```

#### 15.4.9 工具调用质量（Tool Use）

前提：系统采用直连工具提交（Direct Tool Commit），Agent 行为直接落库。

检查：

```text
thought_text 与上下文是否自然连贯？
单次 action 内多次 tool call 的顺序是否正确？
Agent 在调用终止工具后是否还继续执行后续操作？
```

坏例：

```text
Agent 调用 exit_browsing 后又发起了 write_comment。
Agent 在 feed_card 直接 like_post 或 write_comment。
```

### 15.5 评分表

| 维度 | 1 分 | 3 分 | 5 分 |
|---|---|---|---|
| 角色感 | 像分析师 | 部分像用户 | 很像真实用户 |
| 行为自然度 | 固定流程 | 有少量变化 | 行为链真实多样 |
| 人设差异 | 基本无差异 | 有部分差异 | 差异明显且合理 |
| 评论真实感 | 像访谈 | 偶尔真实 | 像真实评论区 |
| 质疑自然度 | 无质疑或乱喷 | 有质疑但单薄 | 质疑自然具体 |
| 心路历程质量 | 像分析报告 | 有时像用户 | 自然解释行为 |
| 工具合法性 | 经常非法 | 偶尔非法 | 基本合法 |
| 报告证据链 | 凭空分析 | 部分有证据 | 证据充分 |

合格线：

```text
平均 >= 4
单项不得低于 3
```

### 15.6 Demo 对比表

每次修改 Prompt 后记录：

```text
Prompt 版本
Demo 名称
观众数
点开人数
收藏人数
评论人数
信息流直接结束人数
广告质疑数
追问具体信息数
thought-only 回合数
ignored tool call 数
人工评分
主要问题
```

报告指标必须区分人数和次数：

```text
主报告转化率只使用按 participant 去重的人数。
评论展示为：评论人数 / 点开人数 + 评论条数。
positiveActionActors 是至少做过点赞 / 收藏 / 评论 / 分享之一的人数，不能把多个行为人数相加。
readSkimActors / readPartialActors / readFullActors 是互斥阅读深度分布，不得作为连续漏斗阶段计算流失。
open_post 重复调用不得重复增加 openCount；exit_browsing 对已结束 journey 不得重复增加 exitCount。
```

### 15.7 坏例库

#### 15.7.1 分析师腔

```text
该内容具有较强的用户痛点洞察，但缺少权威背书。
```

修正方向：

```text
改成用户视角：看着有用，但我有点不确定是不是真的。
```

#### 15.7.2 过度正向

```text
每个用户都点赞、收藏、评论支持。
```

修正方向：

```text
加强低意向、广告敏感、专业挑剔用户权重。
```

#### 15.7.3 评论太正式

```text
请问作者是否可以补充一下具体产品型号和购买渠道？
```

修正方向：

```text
"蹲个型号，别藏着姐妹"
```

#### 15.7.4 工具乱用

```text
feed_card 阶段调用 like_post 或 write_comment
post_detail 阶段调用 open_post
exit_browsing 之后继续 favorite_post
```

修正方向：

```text
工具合法性由 screen-based 校验强制；非法调用由工具服务返回 ok: false / reason 并持久化。
```

### 15.8 版本准入标准

一个 Prompt 版本进入主流程前，需要满足：

```text
[ ] 三个 Demo 都能跑完。
[ ] ignored tool call 占比低于 10%。
[ ] repair 触发率低于 15%。
[ ] 人工评分平均 >= 4。
[ ] 评论真实感 >= 4。
[ ] 不出现严重安全边界问题。
[ ] 报告不输出伪预测率（见 TC-092）。
```
