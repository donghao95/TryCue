# 06 报告生成规格

## 1. 目标

试映报告不是普通 AI 内容点评，而是基于 AI 观众试映证据的 **发布前决策台**。

它要回答：

```text
这版内容能不能发？
最大阻断点在哪里？
哪些观众被打动？
哪些观众没被说服？
最应该保留什么？
下一版先改什么？
改完以后应该重测什么？
```

报告不是：

```text
AI 写的一篇总结文章
普通数据看板
点赞收藏评论统计页
内容优缺点罗列
泛泛的优化建议
真实平台表现预测
```

报告是：

```text
帮助内容创作者判断当前版本是否适合发布、卡在哪里、下一版应该先改什么的决策工具。
```

报告不是终点，而是下一轮试映的入口。目标闭环：

```text
创建内容 → AI 观众试映 → 生成报告 → 判断发布建议 → 定位阻断点 → 修改内容 → 重新试映
```

## 2. 硬性边界

### 2.1 不做精确评分

V1 不做：

```text
87 分
B+
可信度 72
发布潜力 8.6
转化指数 91
```

允许使用：

```text
建议发布 / 修改后发布 / 不建议当前版本发布 / 建议重测
强 / 中等 / 偏弱 / 风险
结论信心：高 / 中 / 低
```

### 2.2 不声称预测真实平台表现

报告中不得出现：

```text
发布后会获得高点赞
真实平台表现预计很好
实际转化率可能达到
这条内容一定会爆
```

允许表达：

```text
在本次 AI 观众试映中……
模拟观众表现显示……
当前试映证据表明……
```

### 2.3 不让 LLM 自由编事实

报告必须基于 Evidence Pack。不允许：

```text
1. 直接把所有 raw events 扔给 LLM 随便总结
2. 让 LLM 自己编人数、比例、分组
3. 没有证据就生成结论
4. 没有评论就说"评论区普遍认为"
5. 没有 low_trust 证据就说"信任风险明显"
```

### 2.4 不把报告写成长篇 Markdown

报告可以有 `summaryMarkdown` 文案，但核心输出必须是结构化 JSON（`ReportOutput`）。前端根据结构化数据渲染卡片，而不是只展示一篇长文。

### 2.5 不平均罗列建议

```text
1 个 P0 必改
1-2 个 P1 建议改
1 个 P2 可选优化
```

### 2.6 不只写问题，也要写保留项

`keepAndChange` 必须同时包含 `keep`（保留项）和 `change`（修改项），至少各 1 条。

## 3. 报告输入

报告生成必须使用结构化证据。输入包括：

```text
标题 (title)
封面图 (coverImageUrl) 和正文图片 (imageUrls)
正文 (bodyText)
agent_journeys / agent_turns / agent_tool_calls / agent_transcript_items
action_logs（作为证据链底层数据，通过观众详情和证据抽屉间接引用）
thought_text / thought_summary
social_interaction_events / social_reactions
simulated_comments / simulated_post_state
run_participants / audience_sampling_directives
read_post 工具调用的 depth（skim / partial / full）和 focus 关键词
exit_browsing 工具调用的结构化字段：reasonCategory / readingDepth / interestLevel / trustLevel
write_comment 工具调用的 intent（ask / doubt / share_experience / agree / joke / pushback）
```

前端用户行为和 Agent 行为默认都进入报告统计：

```text
source = agent_tool
source = human_ui
```

封面分析基于真实封面图像的多模态 AI 视觉观察，不依赖文本描述作为视觉理解的唯一来源。

`runtimeMode=mock` 时使用本地 fallback builder 生成结构完整的 ReportOutput；`runtimeMode=real` 时真实报告模型调用失败必须显式失败，不能静默降级为 Mock/fallback 报告。失败后 run 保持可恢复的 `paused + errorMessage` 状态，由用户选择修复配置后重试或结束流程。

### 3.1 报告页与现场页的数据边界

报告页是最终结论的唯一归属页面。以下内容只在报告页展示，不在现场页展示：

```text
发布建议（recommendation）
发布决策台卡片（verdict / funnel / mainBlocker / segments / diagnostics 等）
修改计划（revisionPlan）
重测计划（retestPlan）
关键发现（keyFindings）
改稿建议（rewriteSuggestions）
```

现场页仅展示实时过程信号（如"收藏价值上升"、"具体型号被追问"），不展示最终结论。

群体数据使用原始计数（"30 位 AI 观众中，12 人点开"），不使用预测率（"预测点击率 40%"）。

## 4. Evidence Pack 结构

报告生成以 **Evidence Pack** 为唯一事实源。Evidence Pack 由代码确定性地从试映数据构建（`buildEvidencePack`），LLM 只能基于它输出结论，不能编造 Evidence Pack 中不存在的数据。

```ts
type EvidencePack = {
  meta: {
    runId: string;
    contentVersionId: string;
    audienceCount: number;
    completedCount: number;
    failedCount: number;
    skippedCount: number;
    wasEndedEarly: boolean;
    evidenceQuality: "low" | "medium" | "high";
    evidenceQualityReason: string;
  };
  content: {
    title: string;
    bodyPreview: string;
    imageCount: number;
    coverImageUrl: string | null;
  };
  funnel: EvidenceFunnel;
  exitAnalysis: EvidenceExitAnalysis;
  commentAnalysis: EvidenceCommentAnalysis;
  thoughtAnalysis: EvidenceThoughtAnalysis;
  segments: EvidenceSegments;
  blockers: EvidenceBlocker[];
  audienceGroups: AudienceGroupAnalysis;
  journeySamples: JourneySample[];
  evidenceIndex: Record<string, EvidenceItem>;
};
```

### 4.1 EvidenceFunnel

漏斗指标区分 **人数（actors，按参与者去重）** 和 **次数（events，按行为次数计）**：

```ts
type EvidenceFunnel = {
  exposedActors: number;      openEvents: number;
  openedActors: number;
  readActors: number;         readEvents: number;
  deepReadActors: number;
  readSkimActors: number;     readPartialActors: number;   readFullActors: number;
  viewedCommentsActors: number;
  likedActors: number;
  favoritedActors: number;
  commentedActors: number;    commentEvents: number;
  sharedActors: number;       shareEvents: number;
  exitedActors: number;       exitEvents: number;
  positiveActionActors: number;
  openRate: number | null;
  readRateAfterOpen: number | null;
  deepReadRateAfterOpen: number | null;
  favoriteRateAfterOpen: number | null;
  commentRateAfterOpen: number | null;
  shareRateAfterOpen: number | null;
  positiveActionRate: number | null;
};
```

文案口径：涉及"X 人阅读了正文"时使用 `readActors`（人数），涉及"X 人次阅读行为"时使用 `readEvents`（次数）。比率全部用人数计算。

### 4.2 EvidenceExitAnalysis

```ts
type EvidenceExitAnalysis = {
  totalExits: number;
  riskExitCount: number;          // reasonCategory ∈ {low_trust, too_ad_like, need_more_evidence}
  riskExitRate: number | null;
  byReasonCategory: Record<ExitReasonCategory, number>;
  byReadingDepth: Record<ExitReadingDepth, number>;
  byInterestLevel: Record<InterestTrustLevel, number>;
  byTrustLevel: Record<InterestTrustLevel, number>;
};
```

### 4.3 EvidenceCommentAnalysis

```ts
type EvidenceCommentAnalysis = {
  totalComments: number;
  byIntent: Record<CommentIntent, number>;   // ask / doubt / share_experience / agree / joke / pushback
  representativeComments: EvidenceComment[];
};
```

### 4.4 EvidenceSegments

4 类人群分组：

```ts
type EvidenceSegments = {
  persuaded: SegmentEvidence;                 // 被打动的人
  interestedButNotConvinced: SegmentEvidence; // 高兴趣低信任的人
  skipped: SegmentEvidence;                   // 直接流失的人
  skeptical: SegmentEvidence;                 // 质疑/反驳的人
};
```

### 4.5 EvidenceBlocker

```ts
type EvidenceBlocker = {
  blockerType: BlockerType;
  severity: "low" | "medium" | "high";
  affectedCount: number;
  summary: string;
  evidenceRefs: EvidenceRef[];
};
```

Blocker 类型（7 种）：

```text
feed_attraction      信息流吸引力不足
opening_retention    点开后开头留存不足
trust_evidence       信任证据不足
action_motivation    行动刺激不足
comment_risk         评论风险偏高
target_mismatch      目标人群未命中
evidence_quality     证据质量不足
```

### 4.6 AudienceGroupAnalysis

```ts
type AudienceGroupAnalysis = {
  groups: AudienceGroupStats[];
  contrastUnexpectedRisk: boolean;
};

type AudienceGroupStats = {
  directiveId: string;
  directiveName: string;
  role: AudienceGroupRole;   // core_target / peripheral_target / contrast / exploratory / unknown
  total: number;
  opened: number;
  favorited: number;
  commented: number;
  shared: number;
  evidenceRefs: EvidenceRef[];
};
```

### 4.7 EvidenceItem 与 EvidenceRef

```ts
type EvidenceItem = {
  id: string;
  type: "thought" | "comment" | "tool_call" | "metric" | "blocker" | "segment" | "group" | "journey";
  title: string;
  content: string;
  participantId?: string;
  raw?: unknown;
};

type EvidenceRef = {
  id: string;             // 必须指向 evidenceIndex 中已存在的 id
  type: EvidenceItem["type"];
  label: string;
  participantId?: string;
};
```

`evidenceIndex` 按 tool call 级别标注 phase（feed / post）：`open_post` 自身标为 feed，后续 `read_post` / `exit_browsing` 等标为 post。

## 5. ReportOutput 结构

报告标识口径：

```text
1. 对外页面和 API 使用 runId 定位报告。
2. 报告数据内部仍保留 reportId，用于证据索引、日志追踪和后续重生成 / 版本化扩展。
3. V1 一个 run 最多一份最终报告，前端主流程不需要保存 reportId。
```

```ts
type ReportOutput = {
  verdict: VerdictCard;                    // 发布建议卡
  funnel: FunnelCard;                      // 关键漏斗卡
  mainBlocker: MainBlockerCard;            // 最大阻断点卡
  audienceGroupAnalysis: AudienceGroupAnalysis;  // 人群分组分析（回填 EvidencePack）
  segments: SegmentCard[];                 // 4 类人群卡片
  diagnostics: DiagnosticCard[];           // 5 个诊断维度卡片
  keepAndChange: KeepAndChange;            // 保留项与修改项
  revisionPlan: RevisionAction[];          // 优先修改计划
  retestPlan: RetestQuestion[];            // 重测计划
  evidenceRefs: EvidenceRef[];             // 顶层汇总证据 id（去重）
  keyFindings?: KeyFinding[];              // 关键发现（固定 3 条）
  rewriteSuggestions?: RewriteSuggestions; // 直接改稿建议
  summaryMarkdown?: string;                // 不超过 200 字的摘要
};
```

数据库字段使用 `report_output_json` / `evidence_pack_json`，API view 层字段使用 `reportOutput` / `evidencePack`（结构化对象，非 JSON 字符串）。

## 6. 发布建议枚举

```ts
type Recommendation =
  | "recommend_publish"
  | "modify_then_publish"
  | "not_recommend_current_version"
  | "recommend_retest";
```

展示文案：

| 值 | 文案 |
|---|---|
| recommend_publish | 建议发布 |
| modify_then_publish | 修改后发布 |
| not_recommend_current_version | 不建议当前版本发布 |
| recommend_retest | 建议重测 |

注意：V1 不存在 `backup_version` 枚举值。Prompt 明确禁止输出 `backup_version`，代码层 `RecommendationSchema` 也只有上述 4 个值。

## 7. 发布建议判断规则

代码层 `recommendFromEvidence(evidencePack)` 基于证据质量生成推荐候选，LLM 只能维持或降级（`publish → modify → not_recommend → retest`），不能升级。若 `evidenceQuality` 为 `low`，必须输出 `recommend_retest`。

### 7.1 建议发布

满足大部分：

```text
1. 核心用户明显愿意点开。
2. 收藏 / 价值反馈较强。
3. 信任质疑较少。
4. 风险评论少且可控。
5. 修改建议偏小修小补。
```

### 7.2 修改后发布

满足：

```text
1. 第一眼吸引或价值感较好。
2. 但存在明确信任、表达、证据或评论风险。
3. 修改后大概率能提升表现。
```

这是 V1 中最常见的建议。

### 7.3 不建议当前版本发布

满足：

```text
1. 大量观众在信息流阶段直接结束浏览。
2. 核心用户也不信任。
3. 高风险质疑集中。
4. 内容价值表达不清。
5. 专业领域结论过满，可能误导。
```

### 7.4 建议重测

满足：

```text
1. 试映结果分歧大。
2. 观众行为异常。
3. 输入内容不完整。
4. Agent 失败率过高。
5. evidenceQuality 为 low。
```

## 8. 5 张核心卡片字段定义

### 8.1 VerdictCard（发布建议卡）

```ts
type VerdictCard = {
  recommendation: Recommendation;
  recommendationLabel: string;
  confidence: "low" | "medium" | "high";
  headline: string;
  oneSentence: string;
  topOpportunity: string;
  topRisk: string;
  priorityFix: string;
  evidenceRefs: EvidenceRef[];
};
```

回答：当前版本建议发不发？最大机会是什么？最大风险是什么？优先改什么？

### 8.2 FunnelCard（关键漏斗卡）

```ts
type FunnelCard = {
  audienceCount: number;
  completedCount: number;
  failedCount: number;
  exposedActors: number;
  openedActors: number;
  readActors: number;
  deepReadActors: number;
  readSkimActors: number;
  readPartialActors: number;
  readFullActors: number;
  viewedCommentsActors: number;
  likedActors: number;
  favoritedActors: number;
  commentedActors: number;
  sharedActors: number;
  exitedActors: number;
  positiveActionActors: number;
  openEvents: number;
  readEvents: number;
  commentEvents: number;
  shareEvents: number;
  exitEvents: number;
  openRate: number | null;
  readRateAfterOpen: number | null;
  deepReadRateAfterOpen: number | null;
  favoriteRateAfterOpen: number | null;
  commentRateAfterOpen: number | null;
  shareRateAfterOpen: number | null;
  positiveActionRate: number | null;
  notes: string;
};
```

前端会渲染 **行为漏斗图**（7 阶段：exposed → opened → read → deepRead → readFull → positiveAction → commented）和 **阅读深度分布图**（readSkim / readPartial / readFull）。

### 8.3 MainBlockerCard（最大阻断点卡）

```ts
type MainBlockerCard = {
  blockerType: BlockerType;
  title: string;
  severity: "low" | "medium" | "high";
  affectedCount: number;
  summary: string;
  diagnosis: string;       // 必须解释为什么这是最大阻断点
  evidenceRefs: EvidenceRef[];
};
```

代码层 `selectMainBlocker(blockers)` 按严重度和受影响人数选出候选，LLM 可维持或改选另一个 Evidence Pack 中存在的阻断点，但必须给出理由。

### 8.4 SegmentCard（人群分组卡）

```ts
type SegmentCard = {
  key: SegmentKey;         // persuaded / interested_but_not_convinced / skipped / skeptical
  name: string;
  size: number;
  percentage: number | null;
  summary: string;
  commonTraits: string[];
  representativeThoughts: EvidenceRef[];
  representativeComments: EvidenceRef[];
  suggestedAction: string;
  evidenceRefs: EvidenceRef[];
};
```

4 类人群：

| key | 名称 | 含义 |
|---|---|---|
| persuaded | 被打动的人 | 产生正向行为（点赞/收藏/评论/分享） |
| interested_but_not_convinced | 高兴趣低信任的人 | 点开阅读但未被打动 |
| skipped | 直接流失的人 | 信息流阶段直接 exit_browsing |
| skeptical | 质疑/反驳的人 | 发表 doubt / pushback 评论 |

前端会渲染 **目标人群命中矩阵**（fit × engagement 2×2 四象限）。

### 8.5 RevisionAction（优先修改计划卡）

```ts
type RevisionAction = {
  priority: "P0" | "P1" | "P2";
  title: string;
  action: string;
  reason: string;
  affectedSegment: SegmentKey | "overall";
  expectedImpact: string;
  retestQuestion: string;
  evidenceRefs: EvidenceRef[];
  impactLevel: "high" | "medium" | "low";
  costLevel: "high" | "medium" | "low";
};
```

最多 1 个 P0、2 个 P1、1 个 P2。`impactLevel` 和 `costLevel` 用于前端渲染 **问题优先级矩阵**（impact × cost 2×2 四象限）。

## 9. 诊断维度卡片

5 个 DiagnosticCard，area 分别为 `feed_attraction` / `reading_retention` / `trust_evidence` / `save_value` / `comment_risk`：

```ts
type DiagnosticCard = {
  area: "feed_attraction" | "reading_retention" | "trust_evidence" | "save_value" | "comment_risk";
  title: string;
  status: "strong" | "medium" | "weak" | "risk";
  finding: string;
  reason: string;          // 机制解释
  suggestedFix: string;
  evidenceRefs: EvidenceRef[];
};
```

| area | 标题 | 回答 |
|---|---|---|
| feed_attraction | 信息流吸引力 | 用户会不会停下来？标题封面有没有让人想点？ |
| reading_retention | 正文留存力 | 点开后能不能留住人？ |
| trust_evidence | 信任证据 | 用户是否相信内容？是否觉得像广告？ |
| save_value | 收藏价值 | 用户觉得内容有没有用？是否值得收藏复看？ |
| comment_risk | 评论风险与机会 | 评论区可能出现什么质疑？ |

`status` 由代码层 `computeDiagnosticStatus(area, pack)` 基于漏斗指标和退出分析确定性计算，LLM 可覆盖但必须给出理由。

## 10. KeepAndChange

```ts
type KeepAndChange = {
  keep: Array<{ item: string; reason: string; evidenceRefs: EvidenceRef[] }>;
  change: Array<{ item: string; reason: string; evidenceRefs: EvidenceRef[] }>;
};
```

至少各 1 条。`keep` 告诉用户什么有效应该保留，`change` 告诉用户什么有问题应该修改。

## 11. RetestQuestion（重测计划）

```ts
type RetestQuestion = {
  question: string;
  hypothesis: string;          // H1/H2/H3 形式的可证伪假设
  testVersionLabel: string;    // A/B/C 版标签
  relatedAction: string;
  metricToWatch: string;
  expectedDirection: string;
};
```

2-4 个重测问题。每个必须包含可证伪假设和测试版本标签，用于下一轮试映的假设验证。

## 12. KeyFinding（关键发现）

```ts
type KeyFinding = {
  finding: string;       // 结论
  evidence: string;      // 证据
  impact: string;        // 影响
  action: string;        // 动作
  evidenceRefs: EvidenceRef[];
};
```

固定 3 条，遵循"结论 → 证据 → 影响 → 动作"结构。三条应分别来自不同信号源（如 blocker、segment、人群匹配度），不要三条都讲同一件事。

## 13. RewriteSuggestions（直接改稿建议）

```ts
type RewriteSuggestions = {
  recommendedTitles: RewriteSuggestionItem[];        // 2-3 条
  recommendedOpening: RewriteSuggestionItem;         // 正文前 3 行
  recommendedCommentPrompt: RewriteSuggestionItem;   // 评论引导
  recommendedTags: string[];                          // 3-5 个标签
  recommendedCoverText?: RewriteSuggestionItem;
  recommendedBodyStructure?: RewriteSuggestionItem;
};

type RewriteSuggestionItem = {
  text: string;
  reason: string;
};
```

直接可复制的改稿建议，不是泛泛的"优化标题"。

## 14. 证据索引

报告中的每个核心判断都必须带 `evidenceRefs`，引用的 id 必须来自 `EvidencePack.evidenceIndex`。引用不存在的 id 视为编造事实。

代码层 `assertNoInventedEvidenceRefs(reportOutput, validEvidenceIds)` 会在 LLM 输出后校验，发现编造 id 直接抛错。

证据链纳入 Agent Prompt / 工具质量改造的新结构化证据：

```text
read_post 工具调用（depth / focus）：作为"阅读正文但未互动"的行为证据
exit_browsing 结构化字段（reasonCategory / readingDepth / interestLevel / trustLevel）：作为离开动机和信任/兴趣水平的证据
write_comment.intent：作为评论意图分类证据
```

### 14.1 证据抽屉

前端通过 `EvidenceDrawer` 展示证据详情。点击任意 `EvidenceRefChip` 可打开抽屉，展示 `EvidenceItem` 的 `type` / `title` / `content` / `participantId`。

## 15. 报告 Prompt 约束

Prompt 版本：`PROMPT_VERSION_REPORT = "report_decision_dashboard_v1"`（legacy `report_generator_v1` 仅用于历史数据 fallback）。

System Prompt 核心约束：

```text
1. 你只能基于 Evidence Pack 输出结论，不能编造 Evidence Pack 中不存在的数据、人数、比例、分组或证据。
2. 你不能输出精确分数。结论信心只能用 高 / 中 / 低 表达。
3. 你不能预测真实平台表现。只能表达"在本次 AI 观众试映中……"。
4. TryCue 是 AI 观众试映模拟系统，报告必须诚实表达"模拟观众 / AI 观众试映"。
5. 每个核心判断都必须带 evidenceRefs，引用的 id 必须来自"可用证据 id 列表"。
6. 建议必须具体可执行，不要输出"优化标题""增强吸引力"这类泛泛之词。
7. recommendation 只能从 recommend_publish / modify_then_publish / not_recommend_current_version / recommend_retest 中选一个，不得输出 backup_version。
8. 代码给出的推荐候选是 X。你可以维持或降级（publish→modify→not_recommend→retest），但不得升级。若 evidenceQuality 为 low，必须输出 recommend_retest。
9. 代码候选最大阻断点是 X。你可以维持或改选另一个 Evidence Pack 中存在的阻断点，但必须给出理由。
```

Prompt 还会显式给出：
- 可用证据 id 列表（来自 `evidenceIndex`）
- 代码给出的推荐候选和最大阻断点候选
- ReportOutputSchema 的字段要求
- keyFindings / rewriteSuggestions 的示例

## 16. 报告生成流程

```text
1. generateReportAndCompleteRun(runId) 入口
2. 从 DB 读取 run / content / postState / journeys / comments / logs / toolCalls / turns / participants / directives
3. buildEvidencePack(input) 确定性构建 EvidencePack
4. recommendFromEvidence(pack) 生成推荐候选
5. selectMainBlocker(pack.blockers) 选择最大阻断点候选
6. 真实模式：generateReportWithLLM({ evidencePack, recommendationCandidate, mainBlocker, imageUrls })
   - LLM 输出 JSON
   - coerceReportOutput(...) 防御性补全
   - ReportOutputSchema.safeParse 校验
   - assertNoInventedEvidenceRefs 校验证据 id
   - assertNoBannedScoreFields 校验无分数字段
   - assertNoRealPlatformClaims 校验无真实平台预测
7. Mock 模式：buildFallbackReportOutput(pack, recommendation, mainBlocker, wasEndedEarly)
   - 调用各 buildFallback* 子 builder
   - ReportOutputSchema.safeParse 校验（保证 mock/real 路径产出结构一致）
8. prisma.report.upsert 持久化 recommendation / reportOutputJson / evidencePackJson / model / promptVersion
9. 推送 run.completed + report 事件
```

### 16.1 报告重生成

`POST /api/runs/:runId/report?regenerate=true` 触发 `regenerate=true` 路径：

```text
1. 仅在 run 已处于 terminal 状态（completed / paused）时允许
2. 不修改 run 状态、clock 或重新 emit run.completed
3. 只更新 report 记录并 emit report.regenerated 事件
4. 失败时不破坏已有报告
```

## 17. 报告质量验收

合格报告必须满足：

```text
[ ] 第一屏有明确发布建议（verdict.recommendation）
[ ] 建议理由引用行为证据（evidenceRefs）
[ ] 5 个诊断维度完整（diagnostics）
[ ] 4 类人群分组完整（segments）
[ ] 最大阻断点明确（mainBlocker）
[ ] 修改计划有优先级（revisionPlan：1 P0 + 1-2 P1 + 1 P2）
[ ] 重测计划有可证伪假设（retestPlan：hypothesis + testVersionLabel）
[ ] 关键发现固定 3 条（keyFindings）
[ ] 改稿建议具体可复制（rewriteSuggestions）
[ ] keepAndChange 至少各 1 条
[ ] 群体汇总用 x/30 表达，不使用预测率
[ ] 不出现真实流量预测
[ ] 不出现无证据结论
[ ] 不出现精确分数
[ ] recommendation 不为 backup_version
[ ] 所有 evidenceRefs 指向 evidenceIndex 中已存在的 id
[ ] 页面保留模拟标识声明
```

### 17.1 模拟标识声明

报告页面必须保留以下声明文案，不可省略或修改：

```text
以下互动为 AI 试映模拟结果，不代表真实平台数据。
```

该声明应出现在报告页面的显著位置（如页面顶部或结论区域附近），确保用户明确知晓数据性质。
