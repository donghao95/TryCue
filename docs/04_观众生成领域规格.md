# 04 观众生成领域规格

本文是 TryCue V1 观众生成领域的完整规格，涵盖统一身份架构、采样计划、Profile 展开、身份生成、结果层管理和试映入场。

---

## 1. 设计原则与统一身份模型

### 1.1 核心判断

观众生成要解决的是"这一场试映应该由哪些反应视角组成"，不是让用户逐个审核中间画像。

```text
用户确认的是整场观众分布。
系统内部再把分布展开成单个 AudienceProfile 采样 slot。
正式互动身份是 User / Agent / PlatformAccount。
RunParticipant 只在开始试映时冻结本 run 的入场快照。
```

V1 所有行为主体统一建模为：

```text
User
  -> Agent
  -> PlatformAccount
```

前端真人用户和 AI 观众 Agent 的点赞、收藏、评论、回复、评论点赞等行为复用同一 runtime interaction service。运行期不再用临时观众对象直接写业务事实表。

### 1.2 Plan-First 流程

```text
AudienceSamplingPlan
  -> AudienceSamplingDirective
  -> AudienceProfile
  -> User / Agent / PlatformAccount
  -> RunParticipant
```

阶段含义：

```text
Plan 阶段：生成 AudienceSamplingPlan 和 AudienceSamplingDirective，用户审核整场分布。
Confirm 阶段：用户确认 plan，后端校验 directive 数量总和。
Expansion 阶段：系统按 directive 展开 AudienceProfile。
Identity 阶段：系统为 profile 创建 User / Agent / PlatformAccount。
Start 阶段：只把 identity_ready profile 冻结为 RunParticipant。
Run 阶段：RunParticipant 驱动 AgentJourney / AgentTurn / AgentToolCall。
```

`AudienceProfile` 是确认后的系统中间产物，不是用户第一步逐个审核或编辑的对象。

## 2. 非目标

V1 不做：

```text
1. 不让用户在第一步审核、增删或编辑单个 AudienceProfile。
2. 不把 directive 存成 JSON 后再解析；directive 必须有独立表。
3. 不把 AudienceProfile 写成完整人物传记；它只表达采样覆盖和去重所需的 slot 约束。
4. 不接真实外部平台账号或授权；平台名只作为模拟上下文注入 prompt。
5. 不为观众 Agent 做真实 DOM 自动化。
6. 不让观众 Agent 主动打分、写 memory 或调用历史工具。
7. 不在结果层修改已经 start 的 RunParticipant 快照。
```

## 3. 生命周期

### 3.1 创建 run

用户提交标题、正文、图片和试映人数。创建 run 后进入观众计划阶段，不立即创建 AudienceProfile。

默认规模：

```text
快速试映：12 人
标准试映：30 人
自定义：1 到 10000 人；超过 100 人前端必须提示成本和等待时间
```

### 3.2 生成采样计划

后端创建 `scope=sampling_plan` 的 `AudienceGenerationJob`，由 provider 生成：

```text
AudienceSamplingPlan.totalCount
AudienceSamplingPlan.dimensionsJson
AudienceSamplingPlan.planMarkdown
AudienceSamplingDirective[]
```

`planMarkdown` 只用于给用户快速浏览采样计划的试映采样设计 brief，说明系统如何理解内容、为什么这样设计观众、确认后会围绕哪些证据运行试映。它需要引用标题、正文、图片或平台上下文里的具体信息点，但不评价被测内容好坏，不复述具体分组、人数或差异轴。结构化事实源是 `audience_sampling_directives`，不是 Markdown。

真实 provider 的采样计划生成必须使用 stream-native frame protocol。模型输出为 NDJSON：一行一个完整 JSON frame。后端按 `text-delta` 增量累积 buffer，只在遇到换行时解析完整 frame，并把可校验 frame 写入 `live_events` 后通过 SSE 推送给前端。前端在同一个 Audience Studio 审阅界面中用骨架屏和 streaming 状态逐步填充计划说明、拆分维度和 directive 卡片。

frame 流只形成 `AudienceSamplingPlanDraftPreview`，不是正式业务事实。未收到并校验 `plan_completed` 前，不写入 `audience_sampling_plans` 或 `audience_sampling_directives`，也不允许用户确认或编辑为正式 plan。生成完成后，后端将累计 preview 编译为 canonical draft，必须通过以下校验才写入正式表：

```text
1. 收到 plan_completed。
2. totalCount 等于请求 count。
3. 至少有一条 complete directive。
4. 每条 complete directive 都有 name / description / quantity / diversityAxes / rationale。
5. directive.quantity 全部为正整数，合计等于 count。
6. planMarkdown 非空，dimensions 可为空但必须是字符串数组。
```

写库成功后发送 `audience.plan.ready`，前端用正式 `AudienceSamplingPlanView` 升级已有预览卡片，而不是清空重画。frame 中的 `key` 只用于生成期稳定映射；正式 `directive.id` 仍由数据库生成。

若模型输出 malformed frame、未知 frame 类型、缺失字段或最终校验失败，后端记录 parser / validation issue，并在 UI 中展示为生成问题；只有最终 canonical draft 校验失败时才将 job 标记为 failed。不得用半成品 preview 写正式表来掩盖失败。

frame 流失败处理：

```text
1. 前端漏收或断线：不影响后端生成。`audience.plan.frame` 已写入 live_events，客户端按 Last-Event-ID / after 回放后重建 preview。
2. 单个 SSE 推送失败：不影响 canonical 生成。事件已持久化，重连后补齐；最终仍以 `audience.plan.ready` snapshot 为准。
3. 模型输出坏 frame、未知 frame、非法字段、缺少 directive_completed 或缺少 plan_completed：后端 accumulator 记录问题，最终 compile 失败，整次 sampling_plan job 标记 failed。
4. 不支持从某个 frame 继续补写，也不做前端字段兜底。用户重试时重新创建 sampling_plan job，替换旧 preview。
5. 若 frame preview 看起来完整但最终 canonical 校验失败，仍以失败处理；preview 永远不能提升为正式事实。
```

采样计划生成 UI 使用 Audience Studio 内的 preview skeleton / streaming card / failed preview / ready plan 四类状态承载生成过程；`audience.plan.progress` 只提供辅助进度信息，结构化展示以 frame preview 为准。

### 3.3 用户审核计划

未确认前，用户可以：

```text
1. 编辑 planMarkdown 或 dimensions。
2. 编辑单个 directive 的 name / description / quantity / diversityAxes / rationale。
3. 删除明确不需要的 directive。
4. 重新生成整份 plan。
5. 通过"优化观众分布"抽屉和 agent 对话，生成结构化修改建议。
```

"优化观众分布"建议不直接写库。前端展示 `add_directive / update_directive / delete_directive` 操作，由用户确认后按顺序调用现有 directive CRUD API。V1 不新增 apply 或 batch API。

不允许在该阶段新增、编辑或生成单个 AudienceProfile。

### 3.4 确认计划

用户确认后，后端在事务中：

```text
1. 校验至少有一条 directive。
2. 校验 directive.quantity 为正整数。
3. 将 directive.quantity 合计同步为 plan.totalCount 和 TestRun.audienceCount。
4. 保留已有 planMarkdown，不在确认时静默改写用户已看到的计划说明。
5. 锁定 plan 和 directive。
6. 创建观众生成 pipeline job，自动流式展开 profile slot 并并发生成人设。
```

确认后的 plan 不再编辑采样结构。若需要重做分布，只能在尚未被运行期引用前显式 replace active plan；已被 RunParticipant / AgentJourney / 行为事实引用时，不做破坏性重排。

### 3.5 流式展开 AudienceProfile 采样 slot

确认 plan 后，观众生成使用单个 pipeline job 承载 profile expansion 与 identity generation。对外仍只有一个 active audience generation job，便于取消、恢复和前端展示；对内分为 profile expansion pool 与 identity generation pool。AudienceProfile 的职责是覆盖和去重，不是完整人设草稿。

```text
AudienceProfile.runId
AudienceProfile.samplingPlanId
AudienceProfile.samplingDirectiveId
AudienceProfile.sampleIndex
AudienceProfile.samplingLabel
AudienceProfile.demographicsJson
```

`samplingLabel` 是采样点展示标签，不是最终人物名字。目标长度 4-12 个中文字符，最多 20 个中文字符。最终人物名字由 persona 生成后写入 `User.nickname`。

`demographicsJson` 是影响内容反应的基础处境提示，六个字段全部填写；无法确定或不影响反应时填"不限定"：

```text
gender: string
ageRange: string
cityTier: string
lifeStage: string
role: string
spendingPower: string
```

完整身份、性格、MBTI 和表达方式由后续 persona 生成。

同一个 directive 下生成的所有 AudienceProfile 必须整体覆盖 `directive.diversityAxes`；不为每个 profile 单独存 `diversityTagsJson`。覆盖检查通过 prompt 测试和生成结果审查完成。

Profile Expander 按 directive + chunk 流式展开。默认 chunk size 为 10；每个 chunk 请求模型生成指定 `chunkStart / chunkCount` 范围内的 slot，并使用 NDJSON frame 输出。后端每解析到一个合法 `profile_completed` frame，就立即创建 `AudienceProfile(profile_only)` 并推送前端。

Profile Expander 的输入不得包含 `existingProfiles` 或其他 directive 已生成 slot。跨 directive 去重应在 Sampling Plan 的 directive 设计中完成，而不是让后续 chunk prompt 依赖已生成结果。chunk prompt 必须包含 `directive.quantity`、`chunkStart`、`chunkCount` 和当前 sample index 范围，要求标签不模板化且覆盖 directive.diversityAxes。

profile expansion pool 默认最多并发 3 个 directive / chunk 生成请求。同一个 directive 的 chunk 按 sampleIndex 顺序推进；不同 directive 可以并发。每个 `profile_completed` frame 必须包含 `sampleIndex`、`samplingLabel` 和完整 demographics。单个 directive 最终创建数量必须等于 `directive.quantity`；若任一 chunk malformed、数量缺失或最终数量不匹配，该 directive 标记 failed。重试 directive 时清理该 directive 下已生成 profiles / identities 后重新展开整组。

支持的 profile expansion frame：

```json
{ "type": "profile_completed", "sampleIndex": 0, "profile": { "samplingLabel": "预算敏感准妈妈", "demographics": { "gender": "女性", "ageRange": "28-35岁", "cityTier": "二线城市", "lifeStage": "孕晚期", "role": "准妈妈", "spendingPower": "预算敏感" } } }
```

### 3.6 并行生成身份

identity generation pool 为 `profile_only` profile 创建正式行为主体，不等待所有 directive 或所有 chunks 完成。任意 `AudienceProfile(profile_only)` 写库后，即可被 identity pool 通过数据库状态原子 claim 并开始生成：

```text
profile_only -> identity_queued -> identity_generating -> identity_ready / identity_failed
```

生成身份会创建：

```text
User                 昵称和头像
Agent                personaJson、来源、保留策略、收藏状态
PlatformAccount      模拟平台账号 actor
```

`Agent.personaJson` 是可运行人设，字段固定为：

```text
profile: string         — 背景小传/故事：稳定的真实生活背景、人生阶段、家庭/工作上下文、消费上下文、过往生活经历、长期习惯
personality: string     — 稳定性格、风险偏好、社交倾向、情绪表达、决策耐心
mbtiType: string        — 必填，必须是 MBTI 16 型之一
responseStyle: string   — 浏览判断风格、互动倾向、在注入的 platformName 上的评论表达习惯
```

即时反应、心路历程、分数和洞察不写入 Agent；它们来自 AgentTurn、AgentToolCall、action log、interaction fact 和报告层分析。

生成成功后，profile 进入 `identity_ready`；失败进入 `identity_failed` 并保留错误原因。

identity generation 默认最多 10 个 profile 并发。每个 profile 单独调用 Persona Generator，单独创建或替换 `User / Agent / PlatformAccount`，单独推送 `audience.identity.ready` 或 `audience.identity.failed`。不得把多个 profile 合并成一次 persona prompt，也不得给 Persona Generator 传入同组其他 profiles/personas 作为去重上下文。

### 3.7 结果层

确认 plan 后、开始试映前，用户处理的是最终身份，不是采样结构：

```text
编辑身份：昵称头像写 User，人设字段写 Agent
收藏身份：写 Agent.favoritedAt
删除未入场 profile：删除 profile，并按清理规则处理未引用身份
重生身份：创建 single_identity job
重试失败身份：创建 single_identity 或 identities job
补充观众：通过结果层 API 创建 profile，再生成 single_identity
```

生成 pipeline 活跃期间，前端按 profile 级状态控制操作：`identity_queued` / `identity_generating` 锁定；`identity_ready` 一经出现即可编辑、收藏或删除；全局生成未完成时可暂时禁用"重新生成人设"，避免与 pipeline job 抢同一 profile。

结果层新增观众不反向修改已确认的采样计划结构；它是试映前补充最终身份的操作。

### 3.8 开始试映

开始试映不调用 LLM，只将 `identity_ready` profile 物化为 `RunParticipant`：

```text
source_profile_id
sampling_directive_id
user_id
agent_id
platform_account_id
display_name_snapshot
avatar_url_snapshot
profile_snapshot_json
agent_snapshot_json
platform_account_snapshot_json
```

`RunParticipant` 是本 run 的冻结快照。后续修改 User 或 Agent 不应改写已开始试映的历史快照。

## 5. 数据事实源

### 5.1 核心对象

| 对象 | 事实范围 | 关键约束 |
|---|---|---|
| `audience_sampling_plans` | 一场 run 的观众分布计划 | `run_id` 唯一，`plan_markdown` 只是展示快照 |
| `audience_sampling_directives` | 采样指令和人数 | 独立表，按 `plan_id + sort_order` 排序 |
| `audience_profiles` | 确认后展开的采样 slot | 通过 `sampling_directive_id + sample_index` 追溯来源，保存 `samplingLabel` 和 `demographicsJson` |
| `users` | 行为主体昵称头像 | AI 观众和前端用户复用 |
| `agents` | AI 人设、来源、收藏和保留策略 | 不保存心路、评分或运行期记忆 |
| `platform_accounts` | 模拟平台 actor | V1 不保存外部授权 |
| `run_participants` | 开始试映时的入场快照 | 不作为长期身份事实源 |
| `audience_generation_jobs` | 观众生成任务、锁和恢复 | 所有 scope 统一走轻量 AI Task Runner |

### 5.2 核心对象字段

**User**

```text
id, nickname, avatar_url, created_at, updated_at
```

**Agent**

```text
id, user_id, source_run_id, source_profile_id
persona_json (profile / personality / mbtiType / responseStyle)
retention_policy, favorited_at, created_at, updated_at
```

**PlatformAccount**

```text
id, user_id, platform, created_at, updated_at
```

**AudienceSamplingPlan**

```text
id, run_id, generation_job_id, total_count, status
plan_markdown, dimensions_json, error_message, confirmed_at
```

**AudienceSamplingDirective**

```text
id, plan_id, sort_order, name, description, quantity
diversity_axes_json, rationale, expansion_status, expansion_error
```

**AudienceProfile**

```text
id, run_id, sampling_plan_id, sampling_directive_id, sample_index
sampling_label, demographics_json, identity_status
generated_user_id, generated_agent_id, generated_platform_account_id
```

**RunParticipant**

```text
id, run_id, source_profile_id, sampling_directive_id
user_id, agent_id, platform_account_id
source, sort_order
display_name_snapshot, avatar_url_snapshot
profile_snapshot_json, agent_snapshot_json, platform_account_snapshot_json
runtime_status
```

### 5.3 行为事实归属

运行期事实使用以下主体字段：

```text
actor_user_id
run_participant_id
platform_account_id
agent_id
```

业务事实表包括：

```text
social_interaction_events
social_reactions
simulated_comments
loaded_comment_pages
action_logs
agent_journeys
agent_turns
agent_tool_calls
```

工具执行器不直接写 UI 状态；它调用 runtime interaction service，在同一提交层完成状态机校验、幂等控制、事务提交和 SSE 事件。

## 6. 不变量

1. V1 固定 `1 run = 1 contentVersion`。
2. 一个 run 同一时间最多有一个 active sampling plan。
3. `planMarkdown` 不是生成和校验事实源；结构化事实源是 plan dimensions 和 directive 字段。
4. 未确认 plan 前不创建 AudienceProfile。
5. 确认 plan 后 directive 锁定；结果层操作不反向改采样结构。
6. confirm 时必须同步 `plan.totalCount`、`TestRun.audienceCount` 和 directive 数量合计。
7. start 只物化 `identity_ready` profile；部分入场必须由请求显式允许。
8. `RunParticipant` 是快照，不因后续 profile、User 或 Agent 编辑而改写。
9. 删除 profile 时，若身份已被 RunParticipant、评论、互动、日志引用，或 Agent 已收藏，则不物理删除身份。
10. 前端用户和 Agent 的点赞、收藏、分享、评论、回复等行为必须复用统一 runtime interaction service。
11. AudienceProfile 是采样 slot；`samplingLabel` 是采样点展示标签，不能承担完整人设职责。
12. `Agent.personaJson.mbtiType` 必填，且必须是合法 MBTI 16 型之一。
13. 报告和洞察基于持久化证据生成，不由 Agent 主动打分或写 memory。

## 7. 身份清理策略

生成身份默认来源于某个 `AudienceProfile`。删除未入场 profile 时：

```text
1. 如果 Agent 已收藏、retention_policy 要求保留，或身份已被 RunParticipant / 评论 / 互动 / 日志引用，则不物理删除身份。
2. 如果未收藏、策略允许删除且没有引用，则可清理 User / Agent / PlatformAccount。
3. 已 start 的 RunParticipant 快照不因来源 profile 删除而改写。
```

收藏身份写入 `agents.favorited_at`。收藏不改变本 run 的采样计划和 directive 配额。

## 9. Provider 与 Prompt

观众生成拆成五类模型任务：

| 任务 | 输出 | 写库方式 |
|---|---|---|
| Sampling Plan Generator | plan fields + directives | service 校验后写 plan / directive |
| Sampling Plan Revision Agent | revision proposal operations | 不写库，用户确认后调用 CRUD API |
| Profile Expander | AudienceProfile sampling slots | service 按 directive 原子创建 |
| Persona Generator | User / Agent / PlatformAccount fields | service 创建正式身份 |
| Seat Revision Agent | 结果层身份修改建议 | 不写库，用户确认后调用结果层 API |

Prompt 只负责生成候选内容和行为风格；事实边界、字段合法性、幂等、事务提交和 SSE 推送都由后端 service 负责。

Prompt 迁移要求：

```text
1. 平台名通过 platformName 注入，生产 prompt 不写死具体平台名。
2. Profile Expander 使用正向任务描述：生成采样 slot，而不是完整画像或人设草稿。
3. Profile Expander 输出 samplingLabel + demographicsJson，不接收 existingProfiles。
4. Persona Generator 基于单个采样 slot 生成 profile / personality / mbtiType / responseStyle，不接收 existingProfiles / existingPersonas。
5. Persona Generator 必须输出合法 mbtiType；displayName 是这个人的名字，不能是采样标签、职业标签或功能标签。
6. prompt 修改必须配套 real prompt 测试脚本验证 JSON 可解析性、字段完整性、平台名注入和 MBTI 合法性。
```

Mock provider 必须能稳定跑通完整流程。Real provider 使用 streaming 生成采样计划时，可以推送公开 `reasoning` 增量和阶段进度；这些事件只用于用户可见的生成过程反馈，不作为 plan 结构化事实源。Real provider 必须校验结构、数量和字段长度；输出不合格时进入失败或 repair，不静默吞错。Real provider 的非流式调用对 429 / rate limit、408、5xx、网络超时或连接中断做短退避重试；JSON 解析、字段缺失、数量不匹配等确定性校验失败不重试。

## 10. 前端行为

前端观众生成页按 run 状态分段：

| 状态 | 页面行为 |
|---|---|
| `planning_audience` | 展示 plan 生成中或 plan 审核 |
| `generating_audience` | 展示 profile expansion / identity generation 进度 |
| `audience_ready` | 展示结果层身份，可开始试映 |
| `running` 及之后 | 展示现场或现场数据回看 |

关键 UI 边界：

```text
1. 第一屏是可用工作台，不是 landing page。
2. plan 审核页展示整场计划、directive 列表和数量校验。
3. "新增人群"不再是内联手填卡片；多分组联动通过"优化观众分布"抽屉。
4. 抽屉建议卡片必须展示 operation、before/after、warnings 和每条应用状态。
5. 结果层展示最终身份和生成状态。identity_ready 卡片展示昵称、persona 元数据和背景摘要，附带收藏、重生、删除行内操作；非就绪卡片展示 demographics 事实和生成入口。编辑抽屉按身份状态分为完整编辑模式和只读+生成模式；MBTI 使用 16 型下拉选择。
6. 结果层"打磨观众人设"同样先生成建议，不直接写库。
7. 现场页保留模拟标识：以下互动为 AI 试映模拟结果，不代表真实平台数据。
```

### 10.1 前端编辑边界

未确认 plan 前：

```text
用户可编辑 AudienceSamplingPlan / AudienceSamplingDirective。
```

确认 plan 后：

```text
用户不可编辑采样结构。
用户可以在结果层编辑 identity_ready 身份：昵称头像写 User，人设字段写 Agent。
用户可以收藏或删除尚未入场的生成结果。
失败 directive / identity 走明确重试入口。
```

PlatformAccount 不暴露产品编辑字段。

```

`audience.plan.reasoning.delta` 已抑制：不再写入 `live_events`，不通过主 SSE 通道推送。`audience.plan.frame` 和 `audience.plan.progress` 仍正常持久化。`audience.plan.frame` 必须携带 `jobId`、`frameSeq`、`previewId` 或 `planId`。

audience_sampling_directives.expansionStatus / expansionError
audience_profiles.identityStatus / identityError
run_logs

```

## 12. 失败与恢复

1. Plan 生成失败：run 回到可恢复状态并展示错误，用户可重新生成。
2. Directive expansion 失败：标记对应 directive failed，可单独重试。
3. Identity 生成失败：profile 保持 `identity_failed`，可重试失败项或单个重生。
4. Job 中断：通过锁过期和 heartbeat 恢复扫描释放或重试。
5. 结果层删除：只删除未入场、未被引用、未收藏且清理策略允许的身份。
6. replaceActive：只允许清理未被运行期引用的被替换 plan / profile / identity；已经被引用时返回冲突。

## 13. 验收要求

改动观众生成时至少验证：

```text
pnpm --filter @trycue/shared typecheck
pnpm --filter @trycue/api typecheck
pnpm --filter @trycue/web typecheck
```

触及持久化、状态机、job、start 或 run participant 时，应运行相关 API 集成测试；可行时运行 `pnpm test:integration`。

核心验收场景：

```text
create run -> generate plan -> edit directive -> confirm -> expand profiles -> generate identities -> edit/regenerate/favorite/delete result -> start -> participants created
planMarkdown 不成为结构化事实源；确认计划不得静默覆盖已有 planMarkdown
AudienceProfile 是短采样 slot，包含 samplingLabel 和 demographicsJson；directive.diversityAxes 在同组 profiles 中被整体覆盖
personaJson 包含 profile / personality / mbtiType / responseStyle，mbtiType 合法且必填
real prompt 测试脚本能对比修改前后输出，并统计 JSON 可解析性、字段完整性、平台名硬编码和 MBTI 合法性
partial audience start 必须显式 allowPartialAudience
SSE 重连能从 live_events.sequence 回放
非 plan-first 观众生成入口不可用
```
