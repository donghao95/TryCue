# 05 前端规格

本文档定义 TryCue V1 前端工作台、观众生成、现场页、详情层、实时事件消费、报告入口、导航路由和按钮设计。

现场页主叙事是"顶部 HUD + 两列主布局（模拟现场 + AI 观众席）"。行动日志和证据链作为右侧二级视图（tab 切换），不作为常驻主区域。

观众生成前端流程采用 plan-first：先审核 AudienceSamplingPlan / AudienceSamplingDirective，确认后系统自动扩展 profile 和身份，结果层只处理最终身份。领域边界见 `04_观众生成领域规格.md`。

## 1. 目标

核心体验：

```text
用户不是得到一份静态 AI 评分报告，
而是看到内容被发布到一个 AI 试映现场：
帖子状态、评论区、AI 观众席和顶部 HUD 随观众行为同步变化。
```

现场页必须始终保留模拟标识（进入 HUD 或现场头部，不使用底部浮条占用主画面）：

```text
以下互动为 AI 试映模拟结果，不代表真实平台数据。
```

## 2. 页面结构

整个产品只有一个核心页面：**AI 试映工作台**。创建内容和观看实时试映在同一页面完成，发布后不发生视觉跳转。

完整路由、路径表、页面跳转、按钮语义和 push/replace 策略见第 15 节「导航与路由设计」。

### 2.1 draft 结构

发布前是内容创建工作台：

```text
左侧：内容填写区
右侧：试映设置、说明和模拟标识
底部或主按钮：发布到 AI 试映场
```

### 2.2 running 结构

发布后进入现场页结构，采用两列主布局（左窄右宽）+ 顶部 HUD：

```text
┌──────────────────────────────────────────────────────────────┐
│ 顶部 HUD：状态标签 / 模拟进度 / 模拟时间 / 异常 chip / 控制按钮  │
├──────────────────────────┬───────────────────────────────────┤
│                          │ 右侧 tab 切换：观众席 / 运行日志     │
│ 左侧窄幅模拟现场           │ ┌─────────────────────────────┐    │
│ clamp(390px,27vw,430px)  │ │ AI 观众席状态矩阵            │    │
│ ┌──────────────────────┐ │ │ 生命周期 + 行为图标           │    │
│ │ 模拟帖（可滚动）      │ │ │ 详情入口 → 抽屉              │    │
│ │ 媒体 / 正文 / 评论区  │ │ └─────────────────────────────┘    │
│ │ 底部评论输入区        │ │ 或                                  │
│ └──────────────────────┘ │ ┌─────────────────────────────┐    │
│                          │ │ 运行日志（tab 切换显示）     │    │
│                          │ │ 生成 / 调度 / 思考 / 异常     │    │
│                          │ └─────────────────────────────┘    │
└──────────────────────────┴───────────────────────────────────┘
```

宽度分配：左侧模拟现场 `clamp(390px, 27vw, 430px)`；右侧 AI 观众席取剩余宽度，最小 `720px`（`minmax(720px, 1fr)`）；整体沿用现场页 `--page-frame-wide` 宽度约束。

页面信息主次：

```text
第一主角：左侧模拟现场（小红书风格模拟帖）
第二主角：右侧 AI 观众席状态矩阵
第三层：顶部 HUD（run 级摘要，非全量指标条）
第四层：观众详情抽屉、证据链、运行日志（tab 二级视图）
```

completed 状态下：现场数据冻结，HUD 显示"试映已结束 · 现场回看"，主操作为查看报告，评论输入区保留位置但禁用，不自动跳转报告页。

## 3. 视觉风格

全站采用浅色纸面结构风格，不使用深色控制台、玻璃拟态仪表盘或营销 landing page 风格；产品 UI 保持信息密度和操作效率。

### 3.1 气质参考

```text
纸面审阅单 / 结构图
克制、可信、可审查
低圆角、硬边界、轻 offset shadow
运行期保留轻游戏 HUD 的状态反馈
```

不是：

```text
深色控制台
赛博朋克数据大屏
传统后台仪表盘
卡通小游戏
玻璃白卡 + 紫色渐变的通用 SaaS 页面
```

### 3.2 色彩

```text
背景：米白纸面底 + 低对比 28px 网格
主卡片：纸色面板，不使用半透明玻璃质感
边界：墨色硬边界，低圆角 8px
阴影：轻 offset shadow，避免大面积模糊阴影
主行动：低饱和绿色
计划 / 说明：棕金色左色条
运行 / 风险：红橙色
身份 / 系统：青绿色
文字：接近墨黑的小面积强调，正文用灰绿色 muted 文本
```

避免大面积深蓝、高饱和霓虹色、过多渐变、装饰光晕和过重模糊阴影。紫色不再作为全站主强调色，只允许作为局部历史状态或运行指标的辅助色。

### 3.3 背景

背景是安静的纸面工作台，不是视觉主角。使用低对比网格来提供结构感；不使用装饰性 orb、渐变光晕、复杂星空或强动态背景。

### 3.4 结构语言

所有主要页面共享以下视觉语言：

```text
外层 shell：纸面网格背景
顶部 HUD / Header：墨色硬边界 + 纸色底 + 5px offset shadow
主面板：8px 圆角、纸色底、墨色或 line 边界
审阅对象 / 列表项：左色条表示对象类型
按钮：绿色主行动；重新规划/删除等风险操作使用红橙或轻危险态
弹窗 / 抽屉：同样使用纸色面板和硬边界，不回到玻璃拟态
```

### 3.5 稳定滚动条 gutter

全局 `html` 设置 `scrollbar-gutter: stable`，使有无纵向滚动条的页面居中基准一致，避免跨页面切换时产生水平漂移。在 macOS overlay scrollbar 模式下无视觉影响，在 Windows 经典滚动条模式下预留固定空间。

### 3.6 页面级布局个性

各页面在统一 `--page-frame-narrow` / `--page-frame-wide` 契约下保留各自布局节奏：

| 页面 | 布局意图 | 说明 |
|---|---|---|
| 新建页 `/` | 不对称工作台 | 左栏编辑卡占主要宽度，右侧栏为设置/说明辅助面板 |
| 设置页 `/settings` | 双列配置台 | 左列为运行模式、API 与模型配置，右列为调用容量控制；中窄屏回落单列 |
| 历史页 `/runs` | 列表扫描 | 窄帧列表，便于快速浏览和操作 |
| 报告页 `/reports/:id` | 阅读分区 | 窄帧分段阅读，hero + grid sections |
| 现场页 `/runs/:id` | 宽帧一致 | 使用 `--page-frame-wide`，两列主布局（左窄右宽）共享同一宽帧 |
| 观众生成页 | 宽帧一致 | 同现场页，使用 `--page-frame-wide` |

## 4. 创建区字段

### 4.1 表单字段

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| title | input | 是 | 标题 |
| coverImage | CoverImageUploader | 是 | 封面图上传 |
| bodyText | textarea | 是 | 正文 |
| screeningScale | segmented control | 是 | quick / standard / custom |
| customAudienceCount | number input | screeningScale=custom 时是 | 自定义观众数，1-10000 |

已删除的字段：

- ~~accountContext~~ — 账号背景
- ~~targetAudience~~ — 目标人群
- ~~focusQuestions~~ — 重点观察问题
- ~~coverDescription~~ — 封面文字描述

### 4.2 默认值

```text
screeningScale = standard
standard = 30 位观众
quick = 12 位观众
customAudienceCount = 60
```

创建页右侧卡片提供"快速试映 / 标准试映 / 自定义观众数"三种选择。选择自定义后在同一卡片内展开数字输入；当 `customAudienceCount > 100` 时显示成本提示，说明观众数较高会显著增加模型调用和等待时间。

### 4.3 ImageUploader

图片上传组件支持：

```text
拖拽上传或点击选择
多张图片预览，最多 9 张
图片可拖拽调整顺序，第一张为封面
点击图片可查看大图，并支持上一张 / 下一张切换
上传进度指示
文件大小与格式校验（上传文件 <= 5MB，支持 jpg/png/webp）
客户端按原始比例把超过 5MB 的图片压缩到 5MB 以内且最长边不超过 1600px；预览和模拟页必须等比完整显示，不裁切图片内容
上传完成后返回图片 URL，存入 imageUrls；第一张同步为 coverImageUrl
开发/验收环境可提供"使用演示内容"快捷按钮；若当前新建页已有任何草稿内容，覆盖前必须二次确认
```

图片用于：

```text
1. 小红书模拟页图片区展示
2. 多模态 AI 视觉分析
```

### 4.4 内容锁定

发布后，表单字段进入只读状态。原因：

```text
Agent 行为、评论和报告证据都基于发布时内容。
运行中修改标题、封面或正文会破坏证据链。
```

### 4.5 首页入口

首页 topbar 提供"历史试映"和"模型设置"两个辅助入口（详细按钮行为见第 15.5 节各页面按钮表）：

```text
历史试映 -> /runs
模型设置 -> /settings
```

历史试映页展示最近创建的 run，允许打开现场、继续观众准备、查看报告或删除 run。V1不提供运行中快照恢复增强能力，也不承诺从任意中断点继续试映；运行中、暂停中、报告生成中状态只能打开现场查看当前状态。删除为硬删除，但 `running` / `pausing` / `report_generating` 以及存在 active audience generation job 的 run 不允许删除。

### 4.6 模型设置入口

模型设置入口进入 `/settings`。

设置页字段：

| 字段 | 控件 | 说明 |
|---|---|---|
| provider | readonly text | V1固定为 openai-compatible |
| runtimeMode | segmented control | 显式选择 Mock 或 Real，运行时不得再由 baseUrl 推断 |
| apiKey | password input | 只写入服务端，不回显 |
| baseUrl | input | OpenAI-compatible base URL，真实模型运行必填；允许本机、私网和局域网地址 |
| models.fast | custom model picker | 从 `/api/settings/llm/models` 获取后选择，也允许手动输入；用于画像展开、人设批量生成、试映中观众行为回合 |
| models.pro | custom model picker | 从同一模型列表选择，也允许手动输入；用于采样计划、优化问答、打磨人设和报告 |
| capacity.mode | segmented control | 调用策略：自动（推荐）或手动 |
| capacity.preset | segmented control | 预设：保守 / 标准 / 高额度 / 自定义；位于 capacity 高级设置内 |
| capacity 状态 | readonly | 当前 RPM、当前并发、最近限流、冷却时间 |
| capacity 高级设置 | collapsible | preset / initialRpm / minRpm / maxRpm / initialConcurrency / minConcurrency / maxConcurrency / maxRetries |

点击 `capacity.preset` 的保守 / 标准 / 高额度时，前端必须立即更新当前未保存表单草稿中的 `initialRpm` / `minRpm` / `maxRpm` / `initialConcurrency` / `minConcurrency` / `maxConcurrency`；点击自定义只切换 preset，不覆盖用户已填写的数值。此操作不自动保存，仍需用户点击主保存按钮后才写入配置。

前端不得把 API key 写入 localStorage、URL、日志或页面可见状态。保存后只展示 `hasApiKey` / `apiKeyMasked`。设置保存成功后应覆盖 `config/llm.local.yaml` 并立即更新当前 API 进程快照。`runtimeMode=mock` 时后续运行必须使用 Mock provider，即使 real 配置字段完整也不得调用真实模型；`runtimeMode=real` 时 `apiKey`、`baseUrl`、`models.fast`、`models.pro` 必须同时填写，否则保存返回校验错误。配置或请求缺少 `runtimeMode` 时不兼容。需要图片理解的任务使用 `models.pro`，不自动降级到 `models.fast`，设置页必须提示 Pro 模型需支持图片输入。

容量控制区域独立于模型基础配置，默认折叠或置于模型字段下方。用户无需填写容量字段即可保存设置；未传 `capacity` 时后端保留已保存值。容量按钮（校准、重置学习结果、应用推荐值）调用独立 API，不影响主保存按钮的 dirty 状态。

模型列表请求由服务端访问 `baseUrl` 的 `/models`。产品约束是不禁止 `127.0.0.1`、`localhost`、私网或局域网地址，以支持本地和内网 OpenAI-compatible 网关；部署到非可信网络时，安全边界应由设置页/API 的访问控制承担。

## 5. 页面状态机

```text
draft → planning_audience → generating_audience → audience_ready → running → pausing → paused → running
                                                                                           → report_generating → completed
```

| 状态 | 说明 | UI 表现 |
|---|---|---|
| draft | 初始状态 | 可编辑表单，现场区为空 |
| planning_audience | 用户点击发布后，生成 `AudienceSamplingPlan` 和 directive | 内容锁定，显示采样计划生成和审核状态 |
| generating_audience | 用户确认 plan 后，系统按 directive 扩展 profile 并生成人设 | 按 directive 显示扩展、身份生成和失败重试进度 |
| audience_ready | 可入场观众身份生成完成，等待开始试映 | 可编辑/收藏/删除结果层身份，显示"开始试映"按钮；HUD 提供重置试映按钮（清除运行数据后回到此状态） |
| running | 试映进行中 | 顶部 HUD、模拟页、观众席实时变化，右侧运行日志通过 tab 切换查看；HUD 控制按钮位常驻，不可用操作使用禁用态 |
| pausing | 暂停过渡态，等待已开始的观众完成完整旅程 | HUD 显示"正在暂停"，报告入口保留为禁用态 |
| paused | 试映已暂停；不再领取新 AgentJourney，已开始的 Agent 完成完整旅程后停止 | HUD 显示继续/重置试映/结束并生成报告按钮 |
| report_generating | 后端生成报告 | 现场数据冻结，运行控制和报告入口保留为生成中/禁用态 |
| completed | 报告已生成 | `/runs/:runId` 展示冻结现场数据回看，HUD 显示"试映已结束 · 现场回看"，主操作为查看报告，评论输入区禁用，不自动跳转报告页；`/reports/:runId` 展示报告结论 |

> **状态合并说明**：当前实现中 `report_generating` 在 VenueHud 层与 `completed` 合并显示（`const completed = status === "completed" || status === "report_generating"`），HUD 不单独展示"生成报告中"按钮态，统一显示为"试映已结束 · 现场回看"。这是有意的设计决策：报告生成通常耗时较短，用户在 `report_generating` 阶段已经可以回看冻结的现场数据，不需要区分两种终态的 HUD。上表中 `report_generating` 的"生成中/禁用态"描述仅用于后端语义和路由分流，前端 HUD 视觉层统一为 `completed` 表现。

系统异常不进入独立 `failed` 状态。自动重试耗尽后，后端返回或推送 `paused + errorMessage`；前端在暂停态展示错误原因和继续、结束并生成报告、重试等操作。

`run.status` 只表示粗生命周期和路由恢复，不表达观众生成细进度。观众生成页的真实进度来自 `AudienceSamplingPlan`、`AudienceSamplingDirective`、active audience generation job 和 profile.identityStatus。

主页面分流规则：

| 状态 | canonical 页面 | 主要操作 |
|---|---|---|
| draft | `/` | 编辑内容，生成 AI 观众席 |
| planning_audience | `/runs/:runId` | 审阅或重新规划采样计划 |
| generating_audience | `/runs/:runId` | 查看身份生成进度，重试失败项；已有可用身份时可选择部分开始 |
| audience_ready | `/runs/:runId` | 审阅生成结果，开始试映 |
| running | `/runs/:runId` | 暂停试映，查看现场 |
| pausing | `/runs/:runId` | 等待暂停完成，禁止新的运行控制 |
| paused | `/runs/:runId` | 继续试映，或结束并生成报告 |
| report_generating | `/runs/:runId` | 查看冻结现场和报告生成中状态 |
| completed | `/runs/:runId` 与 `/reports/:runId` | `/runs/:runId` 回看现场证据；`/reports/:runId` 查看报告；两者允许来回跳转 |

### 5.1 观众生成阶段

用户点击发布后，不应直接进入 running，而是进入采样计划审核和观众生成阶段。

用户点击发布后，前端创建 run 并停留在同一 workbench。随后发送 `POST /api/runs/:runId/audience-sampling-plan` 创建采样计划；前端通过 `GET /audience-sampling-plan`、`GET /audience-generation` snapshot 建立基线，并通过 SSE 局部更新 plan preview、正式 plan、directive、profile 和 identity 状态。

采样计划生成阶段使用完整 Audience Studio 结构：左侧内容预览已锁定，右侧采样计划审阅区以骨架屏和 streaming 状态占位。随着 `audience.plan.frame` 到达，前端逐步填充计划说明、拆分维度和 directive 卡片；`audience.plan.ready` 到达后，用正式 snapshot 将 preview 升级为可编辑审核态。

生成阶段的主界面由 plan preview 内容驱动，而不是由模型内部阶段进度驱动。`audience.plan.reasoning.delta` 已从主 durable `/events` 流中移除，保留用于未来调试通道；主 UI 不再依赖该事件驱动 token 计数或阶段展示。HUD 阶段、页面标题和审阅区标题仍按规划/生成状态展示。收到首个 `audience.plan.frame` 后，右上角规划操作切换为"规划中"，主审阅区展示已生成出的计划说明、维度和 directive preview。`audience.plan.progress` 和 `audience.plan.frame` 驱动 `plan_streaming` 阶段；未来调试通道的 reasoning trace 不得改变持久化 UI 状态。

前端应使用统一的观众准备 UI 状态机承载自动规划、重新规划和确认后的人设生成阶段，而不是让各入口分别维护一组 boolean/ref。推荐细分阶段为：`plan_requesting`（已发起但尚未绑定 job）、`plan_reasoning`（采样计划 job 已开始且尚未进入 frame streaming）、`plan_streaming`（收到首个 `audience.plan.frame`，或 `audience.plan.progress.stage` 进入 `dimensions/directives/quantities/plan_summary`）、`plan_ready`（正式 plan 可审阅）、`plan_failed`、`audience_generating`、`audience_ready`。自动规划和重新规划都进入同一个 `plan_requesting -> plan_reasoning -> plan_streaming -> plan_ready/plan_failed` 流程；区别只在后端 `replaceActive` 分支会做旧 plan、directive、profile 和未引用身份的清理。重新规划创建新 job 时必须重置该 UI 状态机，因此可以重新显示"思考中"。

未确认 plan 前，主路径是通过"优化观众分布"弹窗和 agent 对话，让 AI 基于用户自然语言需求生成结构化修改建议；用户预览建议卡片并点击应用后，才写入 `AudienceSamplingDirective`。每组仍保留编辑和删除按钮，供用户精准修改单个字段或明确删除某组。directive 数量合计允许随用户修改而变化；保存后后端以当前合计同步 `AudienceSamplingPlan.totalCount` 和 `TestRun.audienceCount`。前端在编辑草稿时展示"当前合计 / 目标人数 / 差额"，但差额不是阻断条件。确认前不创建 `AudienceProfile`，也不展示 profile 级结构编辑入口。

确认 plan 后，前端调用 `POST /api/runs/:runId/audience-sampling-plan/confirm`。后端启动观众生成 pipeline：按 directive chunk 流式扩展 `AudienceProfile`，profile 写库后并发生成身份。此阶段 plan / directive 锁定；用户观察 profile / identity 逐个出现和更新，可重试失败 directive 或失败 identity。

`audience_ready` 后展示生成结果层。用户可以编辑已生成身份、收藏或取消收藏身份、删除尚未入场的 profile；这些操作不改变采样计划和 directive 数量。start 只物化 `identity_ready` profile，不调用 LLM。已确认计划后，顶部不再显示"重新规划"作为普通动作；若用户需要回到结构审阅，显示"清空观众"。该动作清空已生成的画像和人设，保留当前观众结构并撤销 plan 确认，回到 `planning_audience`。回到结构审阅后，用户可以继续编辑当前结构，或再点击"重新规划"生成全新结构。

状态流转：draft → planning_audience → generating_audience → audience_ready → running

观众生成阶段页面结构：

┌──────────────────────────────────────────────────────────────┐
│ 顶部 HUD：Audience Studio / 当前阶段 / 重新规划或清空观众 / 确认计划 │
├──────────────────────────────┬───────────────────────────────┤

生成阶段左侧是内容输入预览，应复用现场页小红书模拟帖的视觉壳、图片浏览、标题正文和图片查看能力，让用户在审阅计划时看到接近试映现场的内容形态；但它仍是只读预览，不展示评论区、互动按钮或运行态反馈。

plan 未确认时，右侧审阅区的信息层级必须服务"快速审查整体分布"：

```text
1. 标题只标识区域：采样计划审阅 / 观众生成进度。
2. 顶部使用三个并列面板：观众总数、拆分维度、计划说明。观众总数以当前 directive 列表和未保存草稿的实时合计为主数字，不能优先使用可能滞后的 validation 快照；目标人数弱化为辅助信息；正常状态不重复展示匹配文案和人群数量，仅在未保存编辑或数量差额时显示短提示，保存 directive 后当前合计会成为新的计划总人数。
3. 计划说明位于顶部右侧面板，保持可见，不折叠；正文按审核依据展示，不弱化成备注文本。
4. 拆分维度以 chip 展示，不和总人数重复。
5. sticky 分组工具栏只展示每类 name + quantity/quantity 或 identityReadyCount/quantity，用于一眼看分配并快速跳转；name 为空时必须明确显示未填写，不得 fallback 成"人群 N"。确认前的 AI 辅助入口在顶部 HUD 操作区展示为"问 AI"，打开后弹窗标题为"优化观众分布"；结果层同一位置展示"问 AI"，打开后弹窗标题为"打磨观众人设"。分组工具栏不再承载主操作按钮，也不再提供会展开大段手填表单的"新增人群"主入口。
6. 下方 directive 卡用于详细审核。生成后按分组直接展示紧凑观众名单，不再默认收起；每个观众行使用头像、昵称或 `samplingLabel`、生成状态和轻量操作。已就绪观众展示 Agent 背景 `persona.profile` 的两行摘要，完整长文本可通过延迟 hover / focus 浮窗或编辑抽屉查看；浮窗方向应按当前页面可用空间动态选择，下方空间不足时向上，否则向下，不按卡片序号决定。浮窗不接收鼠标事件，鼠标离开卡片后应消失，避免干扰列表滚动。未生成人设的画像行不触发浮窗，画像详情通过抽屉查看。
```

采样计划 preview 状态规则：

```text
1. preview 只来自 `audience.plan.frame`，不写入正式 plan state。
2. planMarkdown 区域在首个 `plan_markdown_delta` 前显示多行文本 skeleton；收到 delta 后按自然段即时追加。
3. dimensions 区域在首个 `dimension_upsert` 前显示 chip skeleton；收到 upsert 后按 key 去重展示。
4. `directive_started` 创建一张 directive skeleton 卡，使用稳定 key 保持 DOM 和动画连续。
5. `directive_patch` 逐字段填充卡片；缺失字段保留局部 skeleton，不用"人群 N"等伪内容替代。
6. `directive_completed` 且字段完整合法时，卡片状态从 streaming 变为 complete；字段缺失或非法时显示 invalid 状态和问题。
7. `plan_completed` 但尚未收到 `audience.plan.ready` 时，按钮仍不可确认，页面显示"正在校验并写入正式计划"。
8. `audience.plan.ready` 后，preview 卡片按 sortOrder 和临时 key 的视觉位置平滑升级为正式 directive 卡片；从此编辑、删除、确认按钮可用。
9. 若 job failed，保留最后 preview 和错误信息，提供重新生成入口；不得把失败 preview 当成可确认计划。
```

生成期可见按钮：

```text
- 未确认 plan 前显示"重新规划"；已确认 plan 后显示"清空观众"，清空已生成画像和人设并回到结构审阅。
- 确认计划、编辑、删除、问 AI 优化：在正式 `ready_for_review` 前禁用或隐藏，避免用户编辑半成品 preview。
```

生成失败态规则：

```text
1. 若 sampling_plan job failed，前端保留最后 preview 和错误摘要。
2. 失败 preview 仍不可确认、不可编辑、不可删除、不可进入"问 AI 优化"。
3. 主操作只保留"重新生成整份采样计划"；不提供从失败 frame 继续生成，也不提供局部补齐。
4. 错误详情可进入调试/日志区域；主页面只展示简短、可行动的失败原因。
```

directive 审阅卡片结构：

```text
第 N 组
人群短名称 name   目标 X 人
正文三块并列：自然语言 description / 组内差异方向 diversityAxes / 分配理由 rationale
[编辑] [删除]
```

人数应作为标题旁的醒目 pill（例如"目标 5 人"），不要再把 `人群 1` 作为主标题。正文区不重复人群名称和人数，只保留描述、差异方向和分配理由。`sortOrder` 只作为弱排序信息展示为"第 N 组"。操作按钮应降级为轻按钮；删除不应以大红块常驻抢占主审阅视线。

### 5.2 优化观众分布弹窗

"优化观众分布"是未确认 plan 阶段的主修改入口。它不是新增人群表单，而是一个和 agent 多轮对话的弹窗，用于表达"补充某类人、拆分某组、删除低价值人群、重新分配人数、修改某组描述"等意图。当前页面中点击"新增人群"后展开的内联手填卡片应删除，不作为隐藏高级入口保留。

弹窗本地维护对话历史即可，不保存到数据库。关闭弹窗后可以丢弃本地对话；最终只有用户点击"应用到采样计划"后的 plan / directive 事实写入数据库，并通过 `audience.plan.updated` 同步到主页面。

弹窗承载方式：

```text
1. 桌面端使用右侧覆盖式抽屉，不挤压主页面布局，也不使用居中 modal。
2. 抽屉从右侧工作区上方覆盖打开，默认宽度约 560px，建议最小 480px、最大不超过 42vw；窄屏下可放宽到接近全宽。
3. 左侧小红书内容预览应尽量保持可见；抽屉只覆盖右侧采样计划 / 观众席工作区的一部分。
4. 打开抽屉时不要重排 directive 列表、观众列表或页面主布局，避免用户丢失正在看的对象和滚动位置。
5. 背景不使用强遮罩；可使用轻微透明压暗或仅提升抽屉层级。用户应仍能感知原页面上下文。
6. 抽屉内部独立滚动，主页面保持原滚动位置。关闭后恢复到打开前的列表展开态、筛选态和滚动位置。
7. 移动端使用全屏弹层或接近全屏的 bottom sheet，不做多栏并排，也不挤压页面。
8. 后续可增加"展开为全屏工作模式"，用于长对话或复杂批量修改；V1不要求实现。
```

弹窗结构：

```text
┌──────────────────────────────────────────────┐
│ 优化观众分布                              X │
│ 当前 12 人 · 4 个分组 · 未确认计划             │
├──────────────────────────────────────────────┤
│ 对话区                                        │
│ 用户：把 @核心用户 里的预算敏感人群拆出来        │
│ AI：建议新增一组预算敏感用户，并下调原组人数      │
│                                              │
│ 建议卡片                                      │
│ + 新增：预算敏感新手爸妈 2 人                  │
│ ~ 修改：核心用户 人数 5 -> 3                   │
│ - 删除：低意向路人                             │
├──────────────────────────────────────────────┤
│ 输入框：描述想怎么调整，输入 @ 引用分组          │
│ [取消] [应用到采样计划]                       │
└──────────────────────────────────────────────┘
```

交互规则：

```text
1. 输入框支持 @ mention。用户输入 @ 后弹出当前所有分组，可按 name / description / diversityAxes 模糊搜索。
2. 用户选择分组后，前端显示为 @分组名 chip；如果只是手动输入 @xxx 但未从列表选择，不作为 mention。
3. 发送给 agent 的用户消息中，前端在消息末尾附加被 @ 分组的完整结构化上下文，包括 id、name、description、quantity、diversityAxes、rationale、sortOrder。该上下文不在前端消息正文中展示。
4. 多轮对话可以引用历史消息和历史建议卡片，但每轮生成建议时都必须基于当前页面最新 plan snapshot。
5. agent 只生成建议，不直接写库。建议以结构化 operations 表达，前端渲染为 AI SDK message part / tool result 风格的卡片。
6. 每次用户继续追问后，上一条建议卡片不能继续作为默认可应用建议；只有最新有效建议可以应用。
7. 点击"应用到采样计划"后，前端按建议卡片中的 operations 顺序直接调用现有 directive CRUD API。不要新增单独的 apply 或 batch API；如果实现需要复用校验逻辑，应抽后端公共 service 给现有 API 使用。
8. 应用成功或失败的结果要回写到该建议卡片中，而不是只用 toast。
9. 因为逐条调用不是原子事务，卡片必须展示每条 operation 的 applied / failed 状态，并在失败后提示用户可基于最新 plan 继续对话或手动编辑。
10. 删除分组在建议卡片中必须明确展示；若删除后会剩 0 个分组，前端禁止应用，后端也必须拒绝。
```

建议卡片底层语义是结构化 diff，不是纯 UI 文本。对修改某个分组的某些字段，应展示字段级 before / after：

```text
修改：核心用户
人数：5 -> 3
人群描述：原描述 -> 新描述
组内差异方向：新增"预算极敏感"
分配理由：补充为什么将预算敏感用户拆出观察
```

每组的"编辑""删除"按钮继续保留。两条路径职责不同：

```text
优化观众分布：自然语言、多分组联动、需要 agent 判断，适合新增/拆分/合并/重新分配。
单组编辑：用户明确知道要改哪个字段，直接精准修改 name / quantity / description / diversityAxes / rationale。
单组删除：用户明确不需要该组，快速删除并二次确认。
```

## 6. 顶部运行控制 HUD

顶部 HUD 不仅是现场页的状态栏，也承担运行控制功能。两列布局下 HUD 简化为 run 级摘要，不展示全量指标条。

推荐展示：

```text
运行状态标签
模拟进度（finishedCount / totalAudience）
当前模拟时间
异常 chip（skip / fail，仅在非零时显示）
暂停 / 继续
重置试映
查看报告 / 结束并生成报告
```

HUD 控制区应保持固定按钮位。运行中、暂停中、生成报告中、已完成等状态切换时，不应把运行控制或报告入口从 DOM 中移除；当前不可执行的操作使用禁用态、生成中态或已结束态表达。

HUD 控制区包含三个独立操作，职责互不重叠：

```text
1. 暂停 / 继续：运行控制。暂停意味着系统不再 claim 新的 AgentJourney，已开始的 AgentRunner 会完成该 AgentJourney 的完整旅程后停止。
2. 重置试映：破坏性操作。清除运行日志、互动记录、评论、报告和行为计数，保留内容和观众人设，返回 audience_ready 状态。仅在 audience_ready、paused、completed 时可用；running、pausing、report_generating 及其他过渡状态时禁用。
3. 结束并生成报告：仅在 paused/completed 路径可用，不做任何删除。
```

点赞、收藏、评论、分享等细粒度行为计数不再在 HUD 展示，改为通过 SeatCell 行为图标体现（最多 2 个 + `+N`）。行为图标优先级：质疑 > 评论 > 收藏 > 分享 > 点赞 > 点开（质疑影响风险判断，评论信息量最高，收藏代表价值感，分享代表扩散意愿，点赞是弱正反馈，点开是基础前置行为最易被折叠）。生命周期状态（未开始/进行中/已离开/失败）不占用这 2 个行为图标位。质疑属于派生风险信号，不作为 HUD 直接计数展示。

运行中示例：

```text
AI 试映中｜18 / 30｜模拟时间 00:08:42｜跳过 4
[暂停] [查看报告(禁用)]
```

暂停状态下：

```text
AI 试映已暂停｜18 / 30｜模拟时间 00:08:42
[继续试映] [重置试映] [结束并生成报告]
```

completed 状态下：

```text
试映已结束 · 现场回看｜18 / 30｜模拟时间 00:12:05
[重置试映] [查看报告]
```

`paused` 意味着系统不再 claim 新的 AgentJourney，已开始的 AgentRunner 会完成该 AgentJourney 的完整旅程后停止；它不是立即停止所有动作，也不是只完成当前 turn/action。

HUD 只展示过程信号，不展示发布建议、综合评分、爆款概率或预测率。

## 7. 小红书模拟试映页

主体采用小红书网页版内容详情页心智，但不一比一复刻真实平台。两列布局下，左侧模拟现场是垂直滚动卡片。

推荐结构：

```text
作者信息 / 标题
收起态：标题 / 展开入口 / 评论区
展开态：媒体区 / 正文 / 评论区
当天日期
点赞收藏评论
底部评论输入区
```

帖子默认收起，收起态隐藏图片和正文，但保留可滚动评论区；展开后图片、正文、互动栏和评论区作为一条连续帖子整体滚动。completed 状态下评论输入区保留位置但禁用。

图片区域只保留右上角计数、左右切换按钮、左右滑动切换和点击放大，不展示缩略图条、平台红标或额外放大按钮；图片区域需有最大高度限制（当前实现为 `min(54vh, 430px)`），避免上传大图把现场页撑成图片浏览器。作者栏不展示副标签；不为内容强行补话题或地点；正文元信息只展示当天日期。底部评论输入框不展示表情和图片按钮，分享入口放在顶部作者栏。

评论区是现场感核心，应支持：

```text
新评论插入
新评论短暂高亮
评论数 +1
相关观众头像短暂亮起
```

评论回复使用轻缩进和更小头像表达层级，不使用树状分割线或强线程样式，避免偏离小红书移动端评论观感。

新评论插入应表现为新评论从上方轻微浮出，已有评论通过 FLIP 位移动画下移让位。`最新` 排序下新评论进入顶部；`最热` 排序下新评论尊重真实排序，只在实际位置淡入。短时间并发多条评论时合并为一批，评论区标题旁可显示 `+N 条新评论`，避免多个浮层叠加遮挡评论内容。

关键行为可出现轻提示浮层：

```text
陈琳收藏了这篇内容
理性奶爸提出质疑
小雨跳过了内容
阿泽发表了评论
```

限制：

```text
同一时间最多显示 1-2 条行为浮层
1.5-2 秒后自动消失
只展示关键行为，不展示 inspect / 普通阅读等内部动作
```

## 8. AI 观众席

AI 观众席是现场页第二主角，不是日志列表，也不是普通用户列表。

观众详情抽屉是单个观众的行为复盘页，不是画像档案页。默认排序必须先回答“这个人为什么留下/离开/质疑/收藏，以及对内容有什么启发”，再展示支撑证据，最后才是画像上下文。复盘结论和离开判断不得拆成两张重复卡片；退出原因、阅读深度、兴趣和信任等维度应作为紧凑指标并入复盘卡。时间线按最新到最旧展示原子事件，思考和动作分开显示。默认显示最近 5 条；展开时只追加更早记录，不能重复渲染默认已展示的记录。单人详情中动作日志不需要重复展示当前观众姓名。

### 8.1 布局原则

不能按"当前活跃 5 人大卡片 + 其他观众小列表"设计。默认并发可能是 5，但后续最高可能扩展到 20 人活跃。

正确结构：

```text
全部观众座位矩阵
+
状态筛选
+
状态角标
+
点击详情
```

示例：

```text
AI 观众席   当前活跃 26/30

全部 30｜进行中 26｜已点开 22｜已评论 12｜已收藏 8｜已跳过 6｜有质疑 3｜已完成 20｜失败 1

[头像] [头像] [头像] [头像] [头像]
[头像] [头像] [头像] [头像] [头像]
[头像] [头像] [头像] [头像] [头像]
[头像] [头像] [头像] [头像] [+5]
```

活跃人数从 5 到 20 时，只改变头像高亮、呼吸和状态角标，不改变整体结构。生命周期视觉必须能区分 `not_started` 和已离开状态：未开始使用空位/待入场式低强调头像和独立图例，已离开（`finished` / `skipped` / `risk_exit`）使用已参与后的褪色头像，不能让两者看起来一样。

状态筛选项完整集合：全部 / 进行中 / 已点开 / 已评论 / 已收藏 / 已跳过 / 有质疑 / 已完成 / 失败。筛选项服务用户观察行为和风险，不做过多技术状态暴露；`失败` 作为运行状态保留，因为它影响证据完整性和重试判断。

`read_post` 行为在观众席的展示原则：

```text
1. read_post 是"看了正文但未互动"的中间状态，与 open_post 类似不抢眼。
2. audience.action_happened 事件中 read_post 的 animationHint 为 "none"，
   不触发飞行动画、不触发关键行为浮层、不进入 HUD 计数。
3. SeatCell 可以用低强调的"正在阅读 / 扫读 / 已看完"状态文本体现阅读深度，
   但不与点赞、收藏、评论、分享等强互动行为争夺行为图标位。
4. read_post 不修改 post counters，不推送 post_state.updated，
   因此模拟帖数字不会因 read_post 而变化。
```

### 8.3 头像策略

V1 使用开源统一风格头像或基于 seed 的程序化头像。不为每个观众实时生成 AI 头像。

要求：

```text
统一扁平风格
低饱和色
圆形裁切
不太写实
不太卡通
不混用不同来源
```

使用开源头像前必须确认商用、署名、修改和产品 UI 使用许可。

### 8.4 动画原则

```text
静态为主
状态变化时轻动
关键行为时高亮
不要所有观众一直动
同一时间最多 2-3 个显著动画
普通阅读和内部动作不触发飞行动画
read_post 无飞行动画、无高亮、无浮层（animationHint: "none"）
open_post 无飞行动画，仅切换观众状态
```

关键行为可做"观众席 → 模拟帖子"的飞行动画：

```text
收藏：星标飞向收藏按钮，收藏数 +1
评论：气泡飞向评论区，评论区插入新评论
质疑：风险信号上升，在观众席状态、详情层或运行日志中体现，不进入 HUD 直接计数
```

### 8.5 观众生成结果编辑

观众生成阶段应分成三个清晰步骤：

```text
1. 后端任务生成 AudienceSamplingPlan 和 AudienceSamplingDirective，用户审核整场分布。
2. 用户确认 plan 后，后端按 directive chunk 流式扩展 AudienceProfile，并在 profile 可用后自动并发生成人设身份。
3. 结果层展示 identity_ready / identity_failed profile，允许编辑身份、收藏、删除或重试失败项。
```

该阶段不展示运行日志。日志和调度过程属于试映运行阶段；生成阶段只展示 plan、directive、profile 扩展、人设完成度和生成进度。

本版本不提供全局自然语言 command bar，也不提供第一步的 profile 级新增或编辑入口。自然语言修改只存在于"优化观众分布"弹窗内，且必须产出可预览、可应用的结构化建议卡片。所有结构性写操作最终都落到 plan 或 directive API；确认后结构锁定。

在 `planning_audience` 且 plan 未确认时，用户可以：

```text
- 通过"优化观众分布"弹窗让 agent 生成新增、修改、删除 AudienceSamplingDirective 的建议，并在用户确认后应用
- 通过每组编辑按钮精准修改 AudienceSamplingDirective 字段
- 通过每组删除按钮删除明确不需要的 AudienceSamplingDirective
- 重新生成采样计划
- 确认采样计划并进入自动扩展
```

在 `generating_audience` / `audience_ready` 状态，用户可以：

```text
- 查看 directive 扩展和身份生成进度
- 重试失败 directive 的 profile expansion
- 重试失败 identity
- 通过"打磨观众人设"弹窗让 agent 针对某组或某个观众生成结果层修改建议
- 编辑 identity_ready 人设：昵称头像写 User，人设写 Agent，不展示 PlatformAccount 账号资料编辑
- 收藏或取消收藏人设
- 删除尚未入场的生成结果
- 为 profile_only / identity_failed 画像生成或重生人设
```

每个 directive 卡片保留分组展示，用于快速识别人群分布；卡片内展示已扩展 profiles 和生成状态。

结果层身份卡片按 `identityStatus` 分两种布局：

`identity_ready` 卡片：
```text
article.audienceIdentityRow.identity-ready
  button.audienceIdentityMain（点击打开编辑抽屉）
    头像（使用 User.avatarUrl）
    内容区：
      顶行：User.nickname（strong）+ 状态标签
      元数据行：samplingLabel · MBTI · role · lifeStage（灰色小字）
      摘要：Agent.personaJson.profile 的前两行
      hover 提示或编辑抽屉：完整 profile 文本
  div.audienceIdentityActions（垂直居中对齐）
    收藏按钮（Star 图标，favorited 时填充）
    重新生成按钮（RefreshCw 图标，二次确认）
    删除按钮（Trash2 图标，dangerIconButton，二次确认）
```

非就绪卡片（`profile_only` / `identity_queued` / `identity_generating` / `identity_failed`）：
```text
article.audienceIdentityRow.identity-pending.slot-{status}
  button.audienceIdentityMain（点击打开编辑抽屉）
    序号圆圈（sampleIndex + 1，24px 实线边框）
    内容区：
      顶行：samplingLabel（strong）+ 状态标签
      元数据行：role · lifeStage · gender（灰色小字，仅非空且非"不限定"）
      摘要：ageRange · cityTier · 消费能力（单行截断）
      编辑抽屉：完整 demographics 六字段摘要；画像行不触发 hover 浮窗
  div.audienceIdentityActions
    生成/重试按钮（profile_only 或 identity_failed 时显示）
    删除按钮（dangerIconButton）
```

`profile_only` / `identity_queued` / `identity_generating` 是系统生成中状态，不提供画像字段编辑。排队中的 profile 应表达为低干扰占位项，使用轻量状态线和克制 badge；`identity_generating` 可以比 `identity_queued` 更明确地体现进行中，但不应呈现为错误或强警示。`identity_ready` 阶段展示生成出的昵称、头像和人设入口。

观众卡片操作按 profile 级状态控制，而不是按全局 audience generation job 粗粒度禁用：`identity_queued` / `identity_generating` 必须锁定；`identity_ready` 一经出现即可编辑、收藏或删除；`profile_only` / `identity_failed` 展示生成或重试入口。全局 generation job 活跃时可禁用"重新生成人设"，避免和 pipeline job 抢同一 profile。删除和重新生成操作必须通过确认对话框二次确认。

完整性格、MBTI、长期背景和表达方式来自 `Agent.personaJson`。

### 8.6 打磨观众人设弹窗

`generating_audience` / `audience_ready` 阶段的 AI 辅助入口按钮文案统一为"问 AI"，打开后的弹窗标题为"打磨观众人设"。它面向已经生成或正在生成的具体观众结果，用于让 agent 帮用户讨论、解释、修改、重生、删除或新增某组内的具体观众。

"打磨观众人设"复用 5.2 中定义的右侧覆盖式抽屉承载方式：桌面端不挤压主页面、不使用居中 modal，抽屉覆盖右侧观众生成工作区并保留左侧内容预览可见；移动端使用全屏弹层或接近全屏的 bottom sheet。

它和"优化观众分布"共享一套交互框架，但不是同一个业务 agent：

```text
共性：
1. 都是弹窗内本地多轮对话，不保存对话到数据库。
2. 都支持 @ mention、模糊搜索、前端 chip 展示和隐藏结构化上下文。
3. 都让 agent 生成建议卡片，用户预览后应用。
4. 应用时都调用已有用户 API 或普通公开 API，不走 agent-only 写库。
5. 每条 operation 的成功 / 失败状态都回写到建议卡片。

分离：
1. 优化观众分布操作 AudienceSamplingDirective，只能在 plan 未确认前使用。
2. 打磨观众人设操作 AudienceProfile / User / Agent / generated identity，只能在 profile expansion 或 identity generation 之后、start 之前使用。
3. 打磨观众人设可以通过新增观众让对应 directive.quantity、AudienceSamplingPlan.totalCount 和 TestRun.audienceCount 随实时结果增加；修改人群定义和重新规划整场分布仍属于未确认 plan 阶段。
4. 两者 prompt、proposal schema 和后端 suggestion API 分开；前端可以复用同一个 AssistantDialog shell 和 SuggestionCard 渲染框架。
```

`@` 支持两类对象：

```text
@分组：引用一条 AudienceSamplingDirective，并附带该组的完整 directive、当前 profiles、ready / failed / deleted / missing 计数。
@观众：引用一个 AudienceProfile / generated identity，并附带 samplingLabel、demographics、identityStatus、User.nickname / avatarUrl、Agent.personaJson、favorited 状态。
```

输入框 `@` 选择器需要按对象类型分组展示，并支持模糊搜索：

```text
分组：按 directive.name / description / diversityAxes 搜索。
观众：按 samplingLabel / nickname / demographics / personaJson 搜索。
```

建议卡片 operation 类型建议：

```text
update_identity：修改 identity_ready 观众的昵称、头像或 Agent persona 四段，应用时调用 PATCH /audience-profiles/:profileId/identity。
regenerate_identity：重生某个观众身份，应用时调用 POST /audience-profiles/:profileId/identity/regenerate。
delete_profile：删除尚未入场观众，应用时调用 DELETE /audience-profiles/:profileId。
favorite_identity：收藏或取消收藏观众身份，应用时调用 POST /audience-profiles/:profileId/identity/favorite。
retry_identity：重试 identity_failed 观众，应用时调用 retry-identities。
add_profile：为某个已确认分组新增一位观众，应用时创建 AudienceProfile，同步该分组人数、计划总人数和试映观众数，并启动单个人设生成任务。
```

"新增观众"的语义就是新增一个具体观众结果。替换某个观众时使用删除 + 新增，或直接使用重生人设；补齐失败项优先使用重试，用户明确要额外增加时使用新增观众。

开始试映时如果仍有未生成人设的 profile，前端必须提示"已有 X 个观众人设可参与试映，还有 Y 个画像未生成人设，不会参与本次试映"。用户确认后携带 `allowPartialAudience=true`，后端只让 `identity_ready` profile 物化为 RunParticipant 入场。

## 9. 观众详情与洞察详情

### 9.1 观众详情

点击观众头像 / 卡片打开详情抽屉。

抽屉规格：

```text
宽度：480px（响应式回退 min(480px, 94vw)）
位置：fixed right，顶部留 96px 不遮挡全局 HUD
不改变主两列布局，可覆盖右侧观众席一部分

```

### ActionLog 结构化事实表

ActionLog 是统一时间线事实表。每条记录包含：

- `event_kind`：粗类型，`"thought"` | `"tool_call"` | `"system"` | `"exception"`
- `event_payload_json`：结构化原始 payload，包含 `toolName`、`input`、`output`、`content` 等
- `log_text`：fallback 展示文案（中文模板）
- `action`：工具名或 `"thought"`
- `journey_action_id`：对应 AgentTurn；API 投影为 `turnId`，用于调试、分析和后续需要时的分组

展示文案由 view/frontend 从结构化 payload 派生，不再依赖数据库中的中文 `log_text`。前端通过 `timelineActionDisplayText()` 函数根据 `data.toolName` 和 `data.input/output` 生成 i18n 文案。

API 返回原子 timeline 事件；前端详情页也按原子事件展示，不把思考和工具调用合并成一条。

时间线事件类型：

```text
思考：kind="thought"，data.content 为思考文本
动作：kind="tool_call"，data 包含 toolName/input/output
评论：kind="tool_call"，toolName="write_comment"
结果：低强调样式，展示提交结果或计数变化
异常：风险色，展示失败原因或修复提示
```

时间线中的 `read_post` 动作按阅读深度展示行为文案：

```text
skim    快速扫读了正文
partial 认真看了一部分正文
full    基本看完了正文
```

如果 `input.focus` 有值，加"关注：A / B / C"。

`exit_browsing` 展示"离开内容"，辅助信息体现 `reasonCategory`（不相关 / 不感兴趣 / 信任不足 / 广告感太强 / 内容太长 / 证据不足 / 正常看完离开 / 没有更多动作）。

`write_comment` 的 `intent` 在详情层可展示为评论意图标签（提问 / 质疑 / 经验补充 / 认同 / 玩笑 / 反驳）；主评论区不展示 intent，避免污染真实评论观感。

默认隐藏：调度等待、普通结果、内部控制、model raw error。第一版不做"显示调试日志"开关；后续如调试需求明显，再在抽屉内提供该开关，打开后展示完整运行日志、tool result、调度类日志和内部控制事件。

    exitOutcome?: "skipped" | "browsed_and_left" | "risk_exit" | "max_steps";

    // exit_browsing 结构化字段，由 tool output 投影到 journey 视图

    action: string;
    kind: string;           // "thought" | "tool_call" | "system" | "exception"
    data?: {                // 结构化原始事实，前端从此派生展示文案
      toolName?: string;
      input?: Record<string, unknown>;
      output?: Record<string, unknown>;

  }>;
  interactions: Array<{
    type: string;
    simulatedTime: number;
    reason?: string;
  }>;
  comments: Array<{
    commentText: string;
    commentType: string;

### 10.2 前端用户互动

### 10.3 Run 生命周期事件与时钟同步

前端通过 `run.clock.updated` 同步本地时钟状态 `runClock`。时钟由 `clockAnchorAt`（非 null = 走动中，null = 冻结）和 `clockElapsedMs`（已累计毫秒）组成，前端通过 100ms `setInterval` 驱动 HUD 模拟时间显示。

所有 clock anchor 变化或冻结点都必须由后端推送 `run.clock.updated`，包括开始、继续、真正暂停、自然结束进入 `report_generating`、报告完成、reset、retry 和错误冻结。前端收到后直接用事件中的完整 `clock` 覆盖本地状态，不用 `simulatedTime` 秒数重建时钟。

生命周期事件（`run.started`、`run.paused`、`run.resumed`、`run.completed`）只负责 UI 状态和业务快照刷新，不负责时钟校准。自然结束进入 `report_generating` 时，HUD 必须在 `run.clock.updated(reason="report_started")` 到达后立即停止，不等待报告生成完成。

Audience Studio 采样计划生成中页面按中文步骤展示：请求模型、公开推理、拆分维度、观众分组、人数分配、计划说明。步骤含义必须对应已收到的公开推理或 JSON 字段信号，不把最终事务校验伪装成流式解析阶段。公开推理和 JSON 字段解析上下排列；JSON 解析事件最新在上并带轻微进入动画，公开推理在字段解析进入结构化阶段后自动收起但允许用户手动展开。

### 10.5 去重与重连

SSE 断线重连规则：

```text
1. 自动重连，指数退避，最大间隔 30s
2. 重连时携带 Last-Event-ID
3. 服务端基于 live_events.sequence 回放断线期间事件
4. 前端按 eventId 去重
```

## 12. 试映报告

现场页的报告入口常驻。running / pausing 等报告不可用状态下，该入口显示为禁用态。若用户在 paused 状态主动结束试映，入口切换为"结束并生成报告"，前端先调用 `POST /api/runs/:runId/report`；生成成功后仍停留在 `/runs/:runId` 的 completed 现场数据回看态，并把入口切换为"查看报告"。completed 状态下点击"查看报告"时，前端先调用幂等的 `POST /api/runs/:runId/report` 确保报告存在，再进入 `/reports/:runId`。

报告页必须提供"回看现场数据"入口，进入 `/runs/:runId`。completed 的 `/runs/:runId` 不自动跳回报告页，保证用户能在报告结论和现场证据之间来回查看。

报告页路由使用 `runId`，即 `/reports/:runId`。`reportId` 是服务端报告产物的内部主键，只用于调试、证据追踪或日志关联，前端主流程不依赖 `reportId` 跳转。

报告页展示发布建议、五大维度、群体汇总、评论区预演和修改建议。现场页不得展示完整发布结论。

现场页禁止展示预测率或综合评分：

```text
建议修改后发布
综合评分 82 分
预测点击率 73.3%
预测收藏率 23.3%
爆款概率 78%
```

现场页使用原始计数表达：

```text
12 人点开，7 人收藏，5 人评论，4 人跳过。
```

不使用"预测点击率 40%"等预测性包装。页面必须保留模拟标识：`以下互动为 AI 试映模拟结果，不代表真实平台数据。`

报告页仍使用样本表达：

```text
30 位观众中：
22 人愿意点开
13 人产生收藏意愿
9 人想评论或追问
6 人看到封面和标题后直接结束浏览
11 人提到信任不足
```

## 15. 导航与路由设计

本节整合 TryCue V1 前端的 durable URL、页面职责、跨页面跳转、按钮语义和异常路由处理。目标是让用户始终知道自己处在：

```text
新建试映
历史列表
某次试映的准备 / 现场
某次试映的报告
模型设置
```

而不是在多个视觉形态相似但路径语义不清的页面之间来回猜测。

### 15.1 导航原则

#### 15.1.1 URL 只表达稳定位置

URL 表达可恢复、可分享、可刷新后重建的稳定位置，不表达抽屉、弹窗、局部 tab、筛选器、图片查看器等瞬时 UI。

```text
可以进入 URL：
- 新建试映工作台
- 历史试映列表
- 某个 run 的工作台 / 现场
- 某个 run 的报告
- 模型设置

不进入 URL：
- 观众详情抽屉
- 人设编辑抽屉
- 图片查看器
- 删除确认弹窗
- 评论排序
- 观众席筛选
- 运行日志展开 / 收起
- 路径不存在恢复页（用 replace 规范化或显示恢复页）
```

#### 15.1.2 一个 run 只有两个 durable 页面

同一个 `runId` 最多有两个 durable 页面：

```text
/runs/:runId       准备 / 现场 / 现场数据回看页
/reports/:runId    最终报告页
```

`/runs/:runId` 按 `run.status` 分流视觉形态，但语义始终是"这次试映当前事实和证据现场"。`completed` 后仍然可以打开 `/runs/:runId` 回看冻结现场数据、评论、观众席和运行证据；`/reports/:runId` 是基于这些事实生成的结论页。两个页面允许来回跳转。

#### 15.1.3 导航按钮必须说清目标

页面内主按钮和文本按钮默认不使用含糊的"返回"。按钮文案必须表达目标或动作：

```text
返回主页
历史试映
新建试映
模型设置
回看现场数据
查看报告
结束并生成报告
```

"返回主页"是统一的跨页返回入口，目标固定为 `/`。除"返回主页"外，不再使用"返回"文案。设置页的返回按钮也统一为"返回主页"（因为只有主页能进入设置页）。

#### 15.1.4 内部跳转统一走 SPA navigator

应用内部跳转统一走中心导航函数，例如：

```ts
navigateTo(route, { history: "push" | "replace" })
```

不得在内部页面使用裸 `<a href="/runs">` 造成整页刷新。只有外部链接、文件下载、浏览器必须直接处理的资源可以使用原生链接。

### 15.2 路径表

| 路径 | 页面 | 事实源 | 可刷新恢复 | canonical 条件 |
|---|---|---|---|---|
| `/` | 新建试映工作台 | 本地表单草稿 | 恢复本机缓存草稿 | 无 `runId` |
| `/runs` | 历史试映列表 | `GET /api/runs` | 是 | 列表页 |
| `/runs/:runId` | run 准备 / 现场 / 现场数据回看 | `GET /api/runs/:runId` + snapshots + SSE 或静态快照 | 是 | run 存在 |
| `/reports/:runId` | 试映报告 | `GET /api/runs/:runId/report` | 是 | run 已有报告 |
| `/settings` | 设置 | `GET /api/settings/llm` | 是 | 设置页 |

未知路径不应静默显示新建页。推荐行为：

```text
开发阶段：展示"路径不存在"恢复页，提供"新建试映 / 历史试映 / 模型设置"
产品阶段：replace 到 /，并显示一次轻提示"已回到新建试映"
```

### 15.3 push / replace 规则

`push` 用于用户主动进入另一个页面，保留浏览器后退路径。

`replace` 用于规范化、创建后绑定资源、自动状态跳转，避免后退回到已失效或中转状态。

```text
用户点击"历史试映"              push /runs
用户点击"模型设置"              push /settings
用户点击历史项"继续准备"          push /runs/:runId
用户点击"查看报告"              push /reports/:runId
新建 run 成功                    replace /runs/:runId
报告生成成功                     留在 /runs/:runId，并启用"查看报告"
未知路径规范化到 /                replace /
```

### 15.4 返回来源状态

前端不维护全局 `currentRunId` 作为导航目标。跨页面目标必须来自当前 URL、列表项数据或显式传入的返回来源。

设置页的返回目标固定为 `/`（主页）。因为只有主页能进入设置页，不再需要 `returnTo` 机制记录来源。设置页"返回主页"按钮直接跳转 `/`。

其他页面的"返回主页"按钮同样固定跳转 `/`，不需要 `returnTo`。

### 15.9 实现建议

#### 15.9.1 中心导航 API

建议前端引入统一导航助手：

```ts
type NavigateHistoryMode = "push" | "replace";

function navigateTo(route: AppRoute, options?: { history?: NavigateHistoryMode }) {
  const path = pathForRoute(route);
  if (options?.history === "replace") window.history.replaceState(null, "", path);
  else window.history.pushState(null, "", path);
  setRoute(route);
}
```

所有内部按钮只调用该函数或基于该函数封装的 `openHistory/openSettings/openReport/openRun`。

#### 15.9.2 run route helper

历史列表需要基于 run 状态选择主操作目标。`/runs/:runId` 和 `/reports/:runId` 都是合法 durable 页面，completed run 不再自动规范化到报告页。

```ts
function routeForRun(run: { runId: string; status: RunStatus; hasReport?: boolean }): AppRoute {
  if (run.status === "completed" && run.hasReport) return { kind: "report", runId: run.runId };
  return { kind: "workbench", runId: run.runId };
}
```

如果用户点击 completed run 的"现场数据"，直接进入 `{ kind: "workbench", runId }`。

#### 15.9.3 内部 Link 组件

如果需要使用链接外观，应封装内部 Link：

```tsx
<InternalLink route={{ kind: "history" }}>历史试映</InternalLink>
```

组件负责 `preventDefault()` 并调用 `navigateTo()`。不要在应用内部裸用 `<a href="/...">`。

#### 15.9.4 统一布局规范

页面级布局通过 CSS 变量统一管理，定义在 `:root`（base.css）：

```css
--page-frame-narrow: 1180px;  /* narrow 页面视觉外框宽度 */
--page-frame-wide:  1540px;   /* wide 页面视觉外框宽度 */
--page-gutter:      22px;     /* 页面水平内边距 */
```

所有页面 shell 的顶部 padding 统一为 `24px`，确保导航栏距视口顶部距离一致。所有页面 shell 和 AppHeader 的水平 padding 统一使用 `var(--page-gutter)`，保证标题栏左边缘与正文内容左边缘精确对齐，消除 7-8px 无意错位。

AppHeader 宽度按页面类型分两档：

```text
narrow (var(--page-frame-narrow))：新建试映、历史试映、报告、设置
wide   (var(--page-frame-wide))  ：观众生成
```

现场页不使用 AppHeader，VenueHud 宽度与观众生成页一致（wide 档）。VenueHud 控制区最右侧承载"返回主页"按钮。

各页面内容区宽度必须与所属档位的 AppHeader 宽度对齐：

```text
新建试映 createBoard / createEditor / createAside 外框：var(--page-frame-narrow)
历史试映 historyList/historyEmpty：var(--page-frame-narrow)
报告页 reportBoard/drawerLoading/error：var(--page-frame-narrow)
设置页 settingsBoard/settingsPanel 外框：var(--page-frame-narrow)
观众生成 generationHud/generationBoard：var(--page-frame-wide)
现场页 venueHud/venueStage：var(--page-frame-wide)
```

各页面 shell（`.createShell`、`.venueShell`、`.settingsShell`）水平 padding 统一为 `var(--page-gutter)`，确保 AppHeader 边框与正文可见容器边缘视觉对齐。新建页的 `.createBoard` 是正文 frame，不再额外添加水平 inset；`.createEditor` / `.createAside` 的内部 padding 只负责表单内容节奏。

响应式断点通过覆盖 `--page-gutter` 实现：
- `max-width: 1360px`：`--page-gutter: 18px`
- `max-width: 720px`：`--page-gutter: 12px`

设置页是配置工作台，外框对齐 narrow，但正文采用两列：`.settingsRuntimePanel` 承载运行模式、API key/base URL 与模型选择，`.capacityPanel` 承载调用容量控制。表单输入控件（`input`）仍保持 `max-width: 720px`，避免在左列过度拉伸；容量控制里的状态、预设和高级参数在右列内使用 2 列网格。`max-width: 1120px` 以下回落为单列。

## 19. 国际化（i18n）基础设施

前端 UI 文案使用 i18next + react-i18next 做字符串外部化。新增或修改 UI 文案时必须通过 `t(key)` 或 `i18n.t(key)` 引用，不直接硬编码中文；新增 key 必须同步补到 `apps/web/src/locales/zh-CN.ts` 和 `apps/web/src/locales/en-US.ts`。

### 19.1 基础设施文件

```text
apps/web/src/i18n.ts          i18next 初始化（默认语言 zh-CN，fallbackLng: zh-CN，缺 key 时先回退到 zh-CN 资源，zh-CN 仍缺则显示 key 本身）
apps/web/src/locales/zh-CN.ts 简体中文翻译资源（覆盖全部 UI 文案，约 800 行）
apps/web/src/locales/en-US.ts 英文翻译资源，支持中英文界面切换
apps/web/src/i18next.d.ts     key 类型声明（已在 live 分支存在，无需改动）
apps/web/src/main.tsx         入口处 import "./i18n.js" 触发初始化
```

依赖版本：`i18next@^26.3.1`、`react-i18next@^17.0.8`（见 `apps/web/package.json`）。

### 19.2 使用方式分层

不同代码位置使用不同的 i18n 调用方式：

| 位置 | 调用方式 | 示例 |
|---|---|---|
| 函数组件内 | `useTranslation()` hook + `t(key)` | `const { t } = useTranslation(); t("home.title")` |
| 非组件函数（`format.ts`、模块级常量函数等） | `import i18n` + `i18n.t(key)` | `import i18n from "../i18n.js"; i18n.t("status.seat.active")` |
| 类组件（`ErrorBoundary`） | `import i18n` + `i18n.t(key)`（不使用 `withTranslation` HOC） | 同上 |

带变量的字符串使用 i18next 插值：`t("venue.comment.replyCount", { count: n })`，资源文件中写 `"{{count}} 条回复"`。

### 19.4 界面语言切换语义

界面语言切换走设置保存语义，不在点击语言卡片时立即切换：

```text
1. 用户在设置页选择界面语言（zh-CN / en-US）
2. 设置页进入未保存状态（dirty），触发导航守卫
3. 用户点击「保存设置」后调用 i18n.changeLanguage() 并写入当前设备的 localStorage
4. 未保存离开设置页时，界面语言不生效
```

不实现服务端持久化的用户语言偏好，界面语言偏好只保存在当前设备浏览器本地。

### 19.5 非目标

```text
不做服务端持久化的用户语言偏好
不做后端 API 文案 i18n（API 返回的 message / error 仍为开发期约定语言）
不做 packages/shared 的枚举值 i18n（RunStatus / ToolCallStatus 等枚举值不翻译，只翻译面向用户的展示文案）
不引入其他 i18n 框架或更换语言资源格式（保持 i18next + TS 资源文件）
```
